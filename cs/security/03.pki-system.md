# PKI 인증 시스템 (Public Key Infrastructure)

SSL 및 암호기술 이해를 위한 기초 개념. 공개키 신뢰를 위한 인증 기관(CA) 체계, X.509 디지털 인증서 구조, 디지털 서명 검증 방식, 그리고 SSL/TLS와 웹 서비스의 보안 네트워크 구성을 다룬다.

---

## 1. 공개키 신뢰의 필요성

### 1.1 MITM 공격과 검증 필요성

대칭키는 인터넷에서 키를 안전하게 교환하기 어렵고, 비대칭키는 이 문제를 해결한다. 실제 SSL/TLS에서는 효율을 위해 대칭키(세션키)로 통신을 암호화하고, 세션키 자체만 비대칭키로 암호화하여 전달한다. (02.encryption.md 4.1 참조)

그러나 비대칭키 체계에도 치명적인 결함이 있다. PC가 서버로부터 받은 공개키를 검증 없이 그대로 신뢰하면 안 된다.

MITM(Man-in-the-Middle, 중간자 공격) 시나리오

| 순서 | 무슨 일이 일어나는가 |
|------|----------------------|
| 1 | PC와 서버 사이 통신 경로 중간에 공격자가 끼어든다. |
| 2 | 공격자가 자신만의 비대칭키 쌍(공개키 A, 개인키 A)을 만든다. |
| 3 | PC가 서버에 연결을 요청하면, 공격자는 공개키 A를 "서버 공개키"처럼 PC에 전달한다. PC는 이게 진짜 서버 공개키인지 알 수 없다. |
| 4 | PC는 공개키 A로 세션키를 암호화해서 보낸다. 공격자가 이 암호문을 가로챈다. |
| 5 | 공격자는 개인키 A로 복호화하여 세션키를 탈취한다. |
| 6 | 이후 PC와 서버 사이 모든 통신이 세션키로 암호화되는데, 공격자도 세션키를 알고 있으므로 내용을 모두 볼 수 있다. |

따라서 "이 공개키가 진짜 해당 서버의 공개키인가?"를 신뢰할 수 있는 제3자(CA)를 통해 검증해야 한다.

### 1.2 신뢰할 수 있는 제3자: CA

내용증명에서 우체국이 발송 사실을 공적으로 증명하듯, CA(Certification Authority, 인증 기관)는 공개키의 소유자 신원을 공적으로 보증하는 역할을 한다.

| 구분 | 설명 |
|------|------|
| CA의 역할 | 서버(또는 사용자)의 공개키에 신원 정보를 결합하고, CA 자신의 개인키로 디지털 서명하여 인증서를 발급한다. |
| 신뢰 기반 | PC와 서버가 CA를 신뢰한다. OS·브라우저는 VeriSign, Comodo 등 주요 CA와 계약하여, OS 업데이트 시 CA의 공개키(기관 인증서)를 PC에 사전 배포한다. |

키 쌍 생성은 전산 자원을 많이 소모한다. 따라서 특정 유효 기간 동안 사용할 수 있는 인증서를 1회 생성하고, RA(등록 기관)를 통해 CA에서 발급받아 웹 서비스에 등록한다. 이것이 SSL 인증서이다.

## 2. 디지털 인증서 (X.509)

### 2.1 정의 및 목적

디지털 인증서(Digital Certificate)는 공개키와 소유자 정보를 결합하고, CA가 디지털 서명하여 위변조를 방지한 형태이다. 표준 형식은 X.509이다.

| 구성 요소 | 설명 |
|----------|------|
| 사용자/서버 정보 | 도메인명, 조직명, 발급 대상 등 |
| 공개키 | 인증 대상의 공개키 |
| CA 디지털 서명 | 인증서 내용의 해시를 CA 개인키로 암호화한 값. CA 공개키로만 복호화·검증 가능하다. |

HTTPS 통신 시 서버는 공개키를 직접 보내지 않고, 이 X.509 형식의 SSL 인증서를 전달한다.

### 2.2 인증서 발급 과정 (CA가 인증서를 만드는 흐름)

서버 운영자가 SSL 인증서를 받기까지의 단계는 다음과 같다.

| 순서 | 주체 | 동작 |
|------|------|------|
| 1 | 서버 운영자 | 서버용 키 쌍(공개키·개인키)을 생성한다. |
| 2 | 서버 운영자 | RA(등록 기관)에 인증서 발급을 요청한다. 도메인 소유권, 조직 정보 등을 제출한다. |
| 3 | RA | 신원을 확인한 뒤 CA에 발급을 요청한다. |
| 4 | CA | 서버 공개키 + 도메인·조직 정보를 하나의 문서로 묶고, 이 내용을 해시한다. |
| 5 | CA | 해시값을 CA 개인키로 암호화한다. 이 암호화된 값이 CA 디지털 서명이다. |
| 6 | CA | (공개키 + 정보 + CA 서명)을 X.509 형식으로 만들어 인증서를 완성하고, RA를 통해 서버 운영자에게 전달한다. |
| 7 | 서버 운영자 | 인증서와 개인키를 웹 서버(또는 보안 장치)에 설치한다. |

> CA 서명의 의미: "이 인증서의 내용은 CA가 검토했고, 위조되지 않았다"는 CA의 보증이다. CA 개인키로만 서명할 수 있으므로, CA 공개키로 풀어보면 CA가 서명한 것인지 확인할 수 있다.

### 2.3 X.509 인증서 구조

| 필드 | 설명 |
|------|------|
| 버전 (Version) | X.509 버전 (v1, v2, v3) |
| 일련번호 (Serial Number) | CA가 부여한 고유 번호 |
| 서명 알고리즘 (Signature Algorithm) | CA 서명에 사용된 알고리즘 (예: SHA-256 with RSA) |
| 발급자 (Issuer) | 인증서를 발급한 CA 정보 |
| 유효기간 (Validity) | 시작일·만료일. 만료 후에는 갱신 또는 재발급 필요 |
| 주체 (Subject) | 인증서 소유자(도메인, 조직 등) 정보 |
| 공개키 (Subject Public Key Info) | 소유자의 공개키 및 알고리즘 |
| CA 서명 (Signature) | CA 개인키로 암호화된 인증서 내용의 해시 |

> CA의 키 체계가 변경되면, 해당 CA가 발급한 모든 인증서와 기관 인증서를 새로 발급해야 한다.

### 2.4 인증서 검증 과정 (클라이언트가 서버를 신뢰하는 흐름)

사용자가 `https://example.com`에 접속할 때, 브라우저(클라이언트)는 서버가 보낸 인증서를 검증한다. 목적은 "이 서버가 정말 example.com인가? 중간에 위조된 인증서는 아닌가?"를 확인하는 것이다.

| 단계 | 무엇을 하는가 | 왜 필요한가 |
|------|---------------|-------------|
| 1. CA 서명 풀기 | PC에 미리 설치된 CA 공개키로, 인증서에 붙어 있는 CA 서명을 복호화한다. 그 결과로 해시값 A를 얻는다. | CA 개인키로만 만들 수 있는 서명이므로, CA 공개키로 풀리면 "CA가 서명한 인증서"임을 확인할 수 있다. |
| 2. 해시 비교 | 인증서 내용(서명 부분 제외)을 동일한 해시 알고리즘으로 계산한다. 그 결과를 해시값 B라고 한다. A와 B가 같은지 비교한다. | A = B이면 인증서가 CA 서명 이후 위변조되지 않았다는 뜻이다. 누군가 내용을 바꾸면 해시가 달라진다. |
| 3. 유효기간·도메인 확인 | 인증서가 아직 만료되지 않았는지, 접속한 도메인(예: example.com)과 인증서의 주체(Subject)가 일치하는지 확인한다. | 만료되었거나 다른 도메인용 인증서라면 신뢰할 수 없다. |
| 4. 세션키 전달 | 위 검증이 모두 통과하면, 인증서에 들어 있는 서버 공개키를 신뢰한다. 이 공개키로 세션키를 암호화하여 서버에 보낸다. | 이제 서버만 세션키를 복호화할 수 있으므로, 안전한 암호 통신이 시작된다. |

> 인증(Authentication) vs 검증(Verification): 이 문서에서 "검증"은 인증서가 위조되지 않았는지, 유효한 CA가 발급했는지 확인하는 과정이다. "인증"은 그 결과를 바탕으로 "이 서버가 정당한 서버다"라고 판단하는 것을 의미한다. 실무에서는 둘을 구분 없이 "인증"이라고 부르기도 한다.

## 3. PKI 인증 체계 (한국)

### 3.1 인증 기관 계층 구조

한국 공인 인증서 체계는 정책·발급·등록이 계층적으로 구성된다.

| 기관 | 역할 | 비고 |
|------|------|------|
| PAA (Policy Approval Authorities) | 공인 인증서 정책 결정 및 하위 기관 정책 승인 | 과학기술정보통신부 담당 (구 미래창조과학부) |
| PCA (Policy Certification Authorities) | Root CA 발급, 기본 정책 수립 | KISA(한국인터넷진흥원, 구 한국정보보호진흥원). Root CA는 자체 서명 인증서를 사용하며, 모든 인증서의 신뢰 근간이 된다. |
| CA (Certification Authority) | 인증서 발급·취소 등 실질 업무 | 금융결제원(예: YesSign), NCA(한국전산원) 등. 상호 간 신뢰 관계를 맺는다. |
| RA (Registration Authority) | 사용자 신원 확인, CA와의 인터페이스 제공 | 은행·공공기관 등이 RA 역할을 수행한다. |

동일한 CA를 신뢰하는 기관 간에는 인증서를 상호 인정한다. 예를 들어 우리은행(RA)에서 발급받은 인증서를 KB국민은행에서도 사용할 수 있는 이유는, 두 은행이 동일한 CA(예: 금융결제원)를 신뢰하기 때문이다.

### 3.2 공인 인증서와 민간 인증서

| 구분 | 설명 |
|------|------|
| 공인 인증서 | 정부·공공기관이 CA·RA를 담당하여 발급하는 인증서. PC에 저장되는 공개키·개인키 쌍이 공인 인증서이다. 본인 인증·전자 서명 등에 사용된다. |
| 민간 인증서 | 법제도 변경으로 네이버, 카카오, 금융기관 등 민간 기관이 CA·RA가 되어 발급하는 인증서. 인증 체계를 민간에 개방한 것이 우리나라 인터넷 환경 발전에 큰 변화를 가져왔다. |

> SSL 인증서(서버 신원 확인용)와 공인 인증서(본인 인증용)는 목적이 다르다. SSL 인증서는 서버에 설치되어 클라이언트가 서버를 신뢰할 수 있게 하고, 공인 인증서는 PC에 저장되어 사용자 본인을 증명한다.

## 4. 실무 활용

### 4.1 SSL 인증서와 웹 서비스

HTTPS 접속 시 전체 흐름 요약

사용자가 `https://example.com`을 입력했을 때, 아래 순서로 진행된다.

1. 요청 → PC가 서버에 연결을 요청한다.
2. 인증서 전달 → 서버가 자신의 SSL 인증서(공개키 포함)를 PC에 보낸다.
3. 검증 → PC는 2.4의 과정대로 인증서를 검증한다. (CA 서명 유효성, 위변조 여부, 유효기간, 도메인 일치)
4. 세션키 교환 → 검증이 성공하면 PC는 서버 공개키로 세션키를 암호화해 보낸다. 서버는 개인키로 복호화하여 세션키를 얻는다.
5. 암호 통신 → 이후부터는 세션키로 대칭키 암호화 통신을 한다. (02.encryption.md 4.1 하이브리드 방식)

SSL 인증서가 어디에 설치되는지를 파악하면 보안 네트워크 인프라 구조를 분석하는 데 도움이 된다.

| 저장 위치 | 구성 요소 |
|----------|----------|
| 서버/보안 장치 | X.509 형식의 인증서(공개키 포함), 개인키(Private Key). 개인키는 반드시 서버만 보관한다. |

### 4.2 보안 네트워크 인프라

| 구간 | 설명 |
|------|------|
| PC → DMZ | HTTPS 통신. 외부에 노출된 구간이므로 암호화가 필수이다. |
| DMZ | 외부와 내부망 사이의 완충 구간. WAF(웹 방화벽)와 SSL 인증서가 주로 여기에 설치된다. |
| DMZ → 내부망 | HTTP 또는 HTTPS. 내부망도 HTTPS를 구성하면 웹서버, WAS 등에 SSL 인증서가 추가로 필요하다. |

> Windows에서 `certmgr.msc` 명령을 실행하면 인증서 저장소 관리 화면을 볼 수 있다. 여기서 CA 기관 인증서와 신뢰할 수 있는 루트 인증서를 확인할 수 있다.
