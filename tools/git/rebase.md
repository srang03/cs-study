git rebase
git rebase는 새로 만들기이다.
commit에서 변경들을 땐 다음 안쪽에서 시작점을 target 브랜치로 옮겨서 그 시작점 부터 뗀 변경들을 덮는다.

git rebase의 의도
타겟 브랜치의 업데이트가 내 브랜치에 반영되길 원한다.


git merge와 rebase의 차이를 알아야한다.
merge는 타겟 브랜치에서 차이점을 검사하고, 차이 만큼의 코드 변경을 내 브랜치의 HEAD에 얹는다.

충돌나면 한번에 해소하고 merge commit이 하나 생성된다.

rebase는 HEAD가 아니라 안쪽에서 업데이트 한다.
내가 작업한 commit 변경들을 잠깐 분리하고, 타겟 브랜치를 시작점으로 해서 때놨던 commit 변경들을 연결에서 봉합한다.
매 commit 마다 충돌을 해결해줘야 한다.

코드를 놓고 보면 rebase와 merge는 타겟 브랜치의 변경점을 반영한다.로 동일하지만 세부 방식이 다르다.

이 2개는 git 시스템 상에서 큰 차이가 있다.
commit 변경들을 덮는다.
여기서 덮는게 commit을 덮는게 아니라 commit 변경을 덮는다가 포인트이다.
rebase하면 커밋들의 해시 값이 달라진다.
merge는 기존 커밋의 해시를 보존한다.

문제는 깃허브에서 발생함
commit hash가 graph 구성에 중요한 역할을 한다.


리베이스 장점은 그 바뀐 커밋자체가 베이스 브랜치의 변경점을 반영한후의 커밋이되어서, 커밋자체가 완전해짐. 머지는 머지 커밋에서 충돌해결을하는 과정때문에 커밋들이 불완전해짐.
