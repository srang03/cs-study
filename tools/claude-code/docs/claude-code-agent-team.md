# Claude Code 에이전트 팀

> 실험적(experimental) 기능 — 기본 비활성화, 명시적 활성화 필요 (v2.1.32+)

---

## 1. 개요

에이전트 팀을 사용하면 여러 Claude Code 인스턴스가 함께 작동하도록 조율할 수 있다.
한 세션이 팀 리더 역할을 하여 작업을 조율하고, 작업을 할당하며, 결과를 종합한다.
팀원들은 각각 자신의 컨텍스트 윈도우에서 독립적으로 작동하며 서로 직접 통신한다.

단일 세션 내에서 실행되고 메인 에이전트에게만 보고할 수 있는 subagent와 달리, 리더를 거치지 않고 개별 팀원과 직접 상호작용할 수도 있다.

---

## 2. 활성화

환경변수 설정 또는 settings.json에서 활성화:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

---

## 3. 아키텍처

| 구성요소 | 역할 |
|----------|------|
| Team lead | 메인 세션. 팀 생성, 작업 조율, 결과 종합 |
| Teammates | 독립 세션. 각자 컨텍스트 윈도우 보유, 서로 직접 통신 |
| Task list | 팀원이 claim/완료하는 공유 작업 목록. 의존성 지원 |
| Mailbox | 에이전트 간 메시징 시스템 |

- 팀 설정: `~/.claude/teams/{team-name}/config.json`
- 태스크: `~/.claude/tasks/{team-name}/`

---

## 4. 팀 시작

자연어로 태스크와 팀 구성을 설명한다:

```
이 CLI 도구의 설계를 검토할 에이전트 팀을 만들어줘.
UX 담당 1명, 아키텍처 담당 1명, 보안 검토 담당 1명으로 구성.
```

팀원의 모델을 지정할 수 있다:

```
4명의 팀원으로 팀을 만들어줘. 각 팀원은 Sonnet을 사용.
```

**중요**: 팀원은 리더의 대화 기록을 상속하지 않는다. spawn prompt에 태스크에 필요한 충분한 컨텍스트를 포함해야 한다.

---

## 5. 디스플레이 모드

| 모드 | 설명 | 요구 사항 |
|------|------|----------|
| in-process | 모든 팀원이 메인 터미널에서 실행. `Shift+Down`으로 전환 | 없음 |
| split pane | 각 팀원이 별도 pane에 표시 | tmux 또는 iTerm2 필요 |
| auto (기본값) | tmux 세션 안이면 split, 아니면 in-process | 자동 판별 |

설정:

```json
{
  "teammateMode": "in-process"
}
```

또는 CLI 플래그: `claude --teammate-mode in-process`

> split pane은 VS Code 통합 터미널, Windows Terminal에서 지원되지 않는다.

---

## 6. 팀원 상호작용

### in-process 모드

| 조작 | 동작 |
|------|------|
| `Shift+Down` | 팀원 순환 |
| `Enter` | 선택된 팀원의 세션 보기 |
| `ESC` | 팀원 작업 중단 |
| `Ctrl+T` | 태스크 리스트 토글 |
| 텍스트 입력 | 선택된 팀원에게 직접 메시지 |

### 통신 명령

- **message**: 특정 팀원에게 메시지 전송
- **broadcast**: 모든 팀원에게 동시 전송 (토큰 비용 팀 규모에 비례, 남용 주의)

팀원은 작업이 없으면 자동으로 리더에게 idle 알림을 보낸다.

---

## 7. 태스크 관리

- 태스크 상태: Pending → In Progress → Completed
- 태스크 간 의존성 지원 — 선행 태스크 완료 시 자동 해제
- 파일 잠금으로 동시 claim 경쟁 방지
- 팀원은 태스크 완료 후 다음 가용 태스크를 자동 claim

리더에게 대기 요청:

```
팀원들이 태스크를 완료할 때까지 기다려줘
```

---

## 8. 권한

- 팀원은 리더의 권한 설정을 그대로 상속한다
- spawn 시 개별 권한 지정 불가, spawn 후 개별 변경은 가능
- 권한 프롬프트를 줄이려면 spawn 전에 settings.json에서 미리 허용:

```json
{
  "permissions": {
    "allow": ["Bash(npm run *)", "Bash(git *)"]
  }
}
```

---

## 9. Agent Team vs Subagent

| 항목 | Agent Team | Subagent |
|------|-----------|----------|
| 통신 | 팀원끼리 직접 통신 | 리더에게만 보고 |
| 조율 | Task list 기반 자기 조율 | 리더가 모든 작업 조율 |
| 컨텍스트 | 각자 독립 (리더 대화 미상속) | 각자 독립 |
| 토큰 비용 | ~7배 이상 (팀 규모에 비례) | 낮음 (결과만 요약) |
| 적합한 작업 | 병렬 연구, 토론, 교차 계층 작업 | 빠른 단일 집중 작업 |

## 10. 팀 제어

```
# 특정 팀원 종료
researcher 팀원을 종료해줘

# 전체 팀원 종료
모든 팀원을 종료해줘

# 팀 정리
팀을 정리해줘

# 플랜 승인 요구
architect 팀원을 생성하되, 변경 전 플랜 승인을 요구해줘
```

> 팀 정리는 반드시 리더를 통해 수행한다. 팀원이 직접 cleanup하면 팀 컨텍스트가 정상 해제되지 않을 수 있다.

## 11. Hook 이벤트

### TeammateIdle

팀원이 idle 상태에 진입할 때 발생. exit 2로 차단하면 팀원이 stderr 피드백을 받고 작업을 계속한다.

```json
{
  "hooks": {
    "TeammateIdle": [
      {
        "hooks": [{ "type": "command", "command": "~/.claude/hooks/quality-gate.sh" }]
      }
    ]
  }
}
```

### TaskCompleted

태스크가 완료 표시될 때 발생. exit 2로 차단하면 태스크가 완료되지 않고 피드백이 전달된다.

```json
{
  "hooks": {
    "TaskCompleted": [
      {
        "hooks": [{ "type": "command", "command": "if ! npm test; then echo 'Tests failing' >&2; exit 2; fi" }]
      }
    ]
  }
}
```

---

## 12. 베스트 프랙티스

- **팀 크기 3~5명**: 이 범위가 실전에서 가장 효과적
- **spawn prompt에 충분한 컨텍스트**: 팀원은 리더의 대화 기록을 상속하지 않음
- **파일 충돌 방지**: 두 팀원이 같은 파일을 편집하면 덮어쓰기 발생. 팀원별 담당 파일을 분리
- **비용 관리**: 팀원에 Sonnet 사용 고려, 완료 후 즉시 팀 정리
- **처음에는 연구/검토 태스크부터**: 경계가 명확한 작업으로 시작

---

## 13. 제한사항

| 제한 | 설명 |
|------|------|
| 세션 재개 불가 | resume 시 in-process 팀원은 복원되지 않음 |
| 태스크 상태 지연 | 팀원이 완료 표시를 놓칠 수 있음 |
| 종료 지연 | 팀원은 현재 요청/도구 호출 완료 후 종료 |
| 세션당 1팀 | 하나의 세션에서 하나의 팀만 관리 |
| 중첩 불가 | 팀원이 자체 팀을 생성할 수 없음 |
| 리더 고정 | 리더십 이전 불가 |
| split pane 제한 | tmux 또는 iTerm2 필요 (VS Code 터미널 미지원) |
