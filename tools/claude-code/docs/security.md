# Claude Code 보안

권한 아키텍처, 프롬프트 인젝션 방어, 샌드박싱을 다룬다.

---

## 1. 보안 기반

### 권한 기반 아키텍처

Claude Code는 기본적으로 **읽기 전용** 권한으로 동작한다.
파일 수정이나 명령 실행이 필요하면 명시적 승인을 요청한다.

### 내장 보호 기능

| 기능 | 설명 |
|------|------|
| Bash 샌드박싱 | 파일시스템과 네트워크 격리로 승인 프롬프트를 줄이면서 보안 유지 |
| 쓰기 접근 제한 | 실행 디렉토리와 하위 폴더에만 쓰기 가능. 상위 디렉토리 수정 불가 |
| 승인 피로 완화 | 자주 사용하는 안전한 명령을 사용자/코드베이스/조직별로 허용 목록 등록 |
| Accept Edits 모드 | 파일 편집은 일괄 승인하면서 부작용 있는 명령은 승인 유지 |
| 명령 인젝션 탐지 | 의심스러운 bash 명령은 기존 허용 목록과 무관하게 수동 승인 요구 |
| 명령 차단 목록 | `curl`, `wget` 등 웹 콘텐츠를 가져오는 명령은 기본 차단 |

## 2. 프롬프트 인젝션 방어

### 핵심 보호

- **권한 시스템**: 민감한 작업은 명시적 승인 필요
- **컨텍스트 인식 분석**: 전체 요청을 분석하여 잠재적 악의 지침 탐지
- **입력 새니타이징**: 사용자 입력 처리로 명령 인젝션 방지

### 추가 보호

| 보호 | 설명 |
|------|------|
| 네트워크 요청 승인 | 네트워크 요청 도구는 기본 승인 필요 |
| 격리된 컨텍스트 윈도우 | WebFetch는 별도 컨텍스트에서 실행하여 악성 프롬프트 주입 방지 |
| 신뢰 검증 | 첫 코드베이스 실행과 새 MCP 서버에 신뢰 검증 수행 |
| Fail-closed 매칭 | 매칭되지 않는 명령은 기본적으로 수동 승인 요구 |

### 비신뢰 콘텐츠 작업 시 권장 사항

1. 제안된 명령을 승인 전에 검토
2. 비신뢰 콘텐츠를 Claude에 직접 파이프하지 않기
3. 중요 파일 변경 사항 검증
4. 외부 웹 서비스와 상호작용 시 VM 사용 고려
5. 의심스러운 동작은 `/feedback`으로 신고

## 3. 샌드박싱

### 개요

샌드박싱은 OS 레벨 프리미티브를 사용하여 Bash 명령의 파일시스템과 네트워크 접근을 격리한다.
승인 프롬프트를 줄이면서 보안을 유지한다.

### 샌드박싱이 해결하는 문제

| 문제 | 해결 |
|------|------|
| 승인 피로 | 반복 승인으로 주의력 저하 |
| 생산성 저하 | 지속적인 중단 |
| 제한된 자율성 | 승인 대기로 효율 저하 |

> 효과적인 샌드박싱은 파일시스템과 네트워크 격리를 **모두** 필요로 한다. 네트워크 격리 없이는 SSH 키 등의 유출이 가능하고, 파일시스템 격리 없이는 시스템 리소스 백도어가 가능하다.

### OS별 구현

| OS | 기술 | 비고 |
|----|------|------|
| macOS | Seatbelt | 기본 제공, 추가 설치 불필요 |
| Linux/WSL2 | bubblewrap | `apt install bubblewrap socat` 필요 |
| WSL1 | — | 미지원 (커널 기능 부족) |

### 활성화

```
/sandbox
```

### 샌드박스 모드

| 모드 | 설명 |
|------|------|
| Auto-allow | 샌드박스 내 Bash 명령 자동 승인. 샌드박스 외부 접근 시 일반 권한 플로우 |
| Regular permissions | 모든 Bash 명령에 표준 권한 플로우 적용 |

두 모드 모두 동일한 파일시스템/네트워크 제한을 적용한다. 차이는 승인 자동화 여부뿐이다.

## 4. 파일시스템 격리

### 기본 동작

- **쓰기**: 현재 작업 디렉토리와 하위 디렉토리만 허용
- **읽기**: 전체 시스템 읽기 가능 (특정 거부 경로 제외)
- OS 레벨에서 강제하므로 `kubectl`, `terraform`, `npm` 등 모든 서브프로세스에 적용

### 추가 경로 허용

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["/tmp/build", "~/.kube"],
      "denyRead": ["~/.aws/credentials"],
      "allowRead": ["."]
    }
  }
}
```

| 설정 | 설명 |
|------|------|
| `allowWrite` | 추가 쓰기 허용 경로 |
| `denyWrite` | 쓰기 거부 경로 |
| `denyRead` | 읽기 거부 경로 |
| `allowRead` | `denyRead` 영역 내 읽기 재허용 (우선) |

### 경로 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `/` | 파일시스템 루트 절대 경로 | `/tmp/build` |
| `~/` | 홈 디렉토리 기준 | `~/.kube` |
| `./` 또는 접두사 없음 | 프로젝트 루트(프로젝트 설정) 또는 `~/.claude`(사용자 설정) 기준 | `./output` |

> 배열 설정은 모든 scope에서 **병합**된다. 상위 scope가 하위를 교체하지 않는다.

## 5. 네트워크 격리

샌드박스 외부의 프록시 서버를 통해 네트워크 접근을 제어한다.

| 기능 | 설명 |
|------|------|
| 도메인 제한 | 승인된 도메인만 접근 가능 |
| 사용자 확인 | 새 도메인 요청 시 승인 프롬프트 |
| 포괄적 적용 | 모든 스크립트, 프로그램, 서브프로세스에 적용 |

```json
{
  "sandbox": {
    "network": {
      "allowedDomains": ["github.com", "*.npmjs.org"],
      "allowLocalBinding": true
    }
  }
}
```

### 커스텀 프록시

```json
{
  "sandbox": {
    "network": {
      "httpProxyPort": 8080,
      "socksProxyPort": 8081
    }
  }
}
```

## 6. 샌드박스와 권한의 관계

| 계층 | 적용 범위 | 역할 |
|------|----------|------|
| 권한 (Permissions) | 모든 도구 (Bash, Read, Edit, WebFetch, MCP 등) | 도구 사용 가능 여부 판단 |
| 샌드박스 (Sandboxing) | Bash 명령과 서브프로세스만 | OS 레벨 파일시스템/네트워크 격리 |

**심층 방어(Defense-in-depth)로 함께 사용한다:**
- 권한 deny 규칙이 Claude의 접근 시도 자체를 차단
- 샌드박스가 프롬프트 인젝션으로 Claude의 판단이 우회되더라도 OS 레벨에서 차단

## 7. 보안 제한사항

| 제한 | 설명 |
|------|------|
| 네트워크 필터링 | 도메인 기반 제한만 수행. 트래픽 내용 검사 불가 |
| 도메인 프론팅 | `github.com` 같은 광범위 도메인 허용 시 데이터 유출 가능성 |
| Unix 소켓 | `allowUnixSockets`로 Docker 소켓 허용 시 호스트 시스템 접근 가능 |
| 파일시스템 권한 상승 | `$PATH` 내 디렉토리 쓰기 허용 시 코드 실행 가능 |
| Linux 약화 모드 | `enableWeakerNestedSandbox`는 Docker 내 동작을 위한 것이며 보안이 약화됨 |

### 샌드박스 이스케이프 해치

샌드박스 제한으로 명령이 실패하면 Claude가 `dangerouslyDisableSandbox`로 샌드박스 밖에서 재시도할 수 있다. 이 경우 일반 권한 플로우를 거친다.

비활성화:

```json
{
  "sandbox": {
    "allowUnsandboxedCommands": false
  }
}
```

## 8. 클라우드 실행 보안

웹 기반 Claude Code 사용 시 추가 보안 제어가 적용된다.

| 기능 | 설명 |
|------|------|
| 격리된 VM | 각 클라우드 세션이 Anthropic 관리 VM에서 격리 실행 |
| 네트워크 접근 제어 | 기본 제한, 특정 도메인만 허용 가능 |
| 자격증명 보호 | 보안 프록시를 통한 인증. 샌드박스 내 스코프 제한 자격증명 사용 |
| 브랜치 제한 | git push는 현재 작업 브랜치로만 제한 |
| 감사 로깅 | 모든 작업 기록 |
| 자동 정리 | 세션 완료 후 환경 자동 종료 |

Remote Control 세션은 다르게 동작한다: 웹 인터페이스가 로컬 머신의 Claude Code 프로세스에 연결된다. 모든 코드 실행과 파일 접근은 로컬에서 이루어지며, 클라우드 VM이나 샌드박싱은 관여하지 않는다.

## 9. MCP 보안

Claude Code는 MCP 서버 구성을 허용하지만, Anthropic은 제3자 MCP 서버의 정확성과 보안을 검증하지 않는다. 자체 MCP 서버를 작성하거나 신뢰할 수 있는 제공자의 서버를 사용한다.

> **Windows WebDAV 보안 위험**: Windows에서 Claude Code 실행 시 WebDAV 활성화나 `\\*` 경로 접근을 허용하지 않는다. WebDAV는 원격 호스트로의 네트워크 요청을 트리거하여 권한 시스템을 우회할 수 있다.

## 10. 베스트 프랙티스

### 민감 코드 작업

1. 모든 제안 변경을 승인 전에 검토
2. 민감 레포에 프로젝트별 권한 설정 사용
3. 추가 격리가 필요하면 devcontainer 사용
4. `/permissions`로 정기적으로 권한 설정 감사

### 팀 보안

1. Managed 설정으로 조직 표준 적용
2. 승인된 권한 설정을 버전 관리로 공유
3. `ConfigChange` 훅으로 세션 중 설정 변경 감사/차단
4. OpenTelemetry 메트릭으로 사용 모니터링

### 취약점 신고

1. 공개적으로 공개하지 않는다
2. [HackerOne](https://hackerone.com/anthropic-vdp/reports/new?type=team&report_type=vulnerability)로 신고
3. 상세한 재현 단계를 포함한다

## 11. 오픈소스 샌드박스

샌드박스 런타임은 오픈소스 npm 패키지로 제공된다. 자체 에이전트 프로젝트나 다른 프로그램의 샌드박싱에도 사용할 수 있다.

```bash
npx @anthropic-ai/sandbox-runtime <command-to-sandbox>
```

소스 코드: [GitHub](https://github.com/anthropic-experimental/sandbox-runtime)
