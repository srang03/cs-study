# 네이티브(C/C++)와 Java 차이

## 1. 실행 파일이 만들어지는 과정(개념)

### 1.1 네이티브(C/C++)

소스 코드가 OS/플랫폼이 바로 실행할 수 있는 기계어로 번역되고, 오브젝트 파일과 라이브러리(컴파일 산출물)들이 링커(linker)에 의해 결합되어 실행 파일/라이브러리가 만들어진다.

- 전처리(Preprocess): `#include`, 매크로 확장 등 텍스트 치환/삽입
- 컴파일/어셈블(Compile/Assemble): `.c/.cpp` → 오브젝트 파일(`.o/.obj`)
- 링크(Link):
  - 여러 오브젝트/라이브러리를 결합해 실행 파일(`.exe`) 또는 라이브러리(`.dll`, `.so`, `.a`)를 생성
  - Windows에서는 오브젝트 파일은 보통 COFF 계열, 최종 실행 파일/라이브러리는 PE 형식을 사용한다(예시).

### 1.2 Java

자바는 소스가 OS 기계어로 바로 컴파일되어 최종 실행 파일로 만들어지는 방식이 아니라, 먼저 바이트코드(`.class`)로 컴파일되고 JVM이 이를 로딩/검증/해석(필요 시 JIT)하여 실행한다.

- 빌드 타임: `.java` → `.class` (컴파일)
- 런타임: JVM이 필요한 클래스를 ClassLoader로 로딩하고, 링크에 해당하는 작업을 수행한 뒤 실행
  - `Linking`: Verify → Prepare → Resolve
    - Verify: 바이트코드가 JVM 규칙을 만족하는지 검증
    - Prepare: 정적 필드 메모리 할당 및 기본값 설정 등 실행 준비
    - Resolve: 심볼릭 레퍼런스를 실제 참조 대상으로 해석(필요 시)
  - `.class`에는 대상 타입/필드/메서드를 가리키는 심볼릭 레퍼런스가 포함될 수 있으며, 실제 참조 대상 해석은 런타임에 수행될 수 있다.
  - 이 때문에 클래스/의존성을 못 찾는 문제는 빌드 타임이 아니라 런타임에 드러날 수 있다.

## 2. 링킹(네이티브) vs 클래스 해석(자바): 정적/동적 개념 정리

### 2.1 정적 링킹 / 동적 링킹(네이티브 관점)

- 정적 링킹(static linking): 필요한 라이브러리 코드를 빌드 시점에 실행 파일에 포함한다(예: `.a`, `.lib` 사용)
- 동적 링킹(dynamic linking): 라이브러리를 별도 파일로 유지하고, 실행 시 로더가 적재하여 결합한다(예: `.so`, `.dll`)

### 2.2 자바의 “링크”

자바는 C/C++의 “링커 단계”가 별도로 존재하지 않고, 이에 해당하는 작업(검증/준비/해석)이 JVM 런타임(ClassLoader의 Linking)에서 수행된다.

- `.class`는 라이브러리(정적/동적) 가 아니라 JVM이 실행하는 바이트코드 산출물이다.
  - 배포 단위는 보통 `.jar`이며, JVM은 지정된 경로에서 클래스를 찾아 로딩한다.
    - 클래스패스(classpath): 클래스를 찾는 검색 경로
    - 모듈패스(module path): JPMS 모듈을 찾는 검색 경로(모듈을 사용할 때)

## 3. 코드 조직화: namespace / package / (module)

- C++ `namespace`: 언어 기능으로 이름 충돌을 방지하기 위한 코드 수준의 묶음
  - 빌드 산출물(라이브러리/실행 파일)과는 별개의 개념
- Java `package`: 클래스의 이름 공간(namespace) 이자 접근 제어/구조화를 위한 묶음
  - 보통 디렉토리 구조와 매핑되며, `import`는 패키지의 이름 해석을 돕는다.
- (추가) Java `module`(JPMS, Java 9+): 패키지보다 큰 배포/캡슐화 단위
  - `module-info.java`로 공개/의존성을 선언하며, 모듈패스로 로딩될 수 있다.

## 4. `#include` vs `import`(전처리 유무)

- `#include`(C/C++): 전처리 단계에서 파일 내용을 소스에 삽입(텍스트 치환)
  - 헤더 포함 방식/중복 포함 방지(`#pragma once`, include guard)가 중요해진다.
- `import`(Java): 전처리로 코드를 “붙이는” 것이 아니라, 컴파일러가 이름(심볼) 해석을 할 때 참조를 편하게 해주는 문법
  - 실제 바이트코드 결합은 JVM의 클래스 로딩/해석(런타임)으로 이어진다.

## 5. 포인터/참조, 메모리 모델의 관점 차이

### 5.1 포인터 vs 레퍼런스

- C 포인터(pointer): 주소를 값으로 다루며 산술 연산/임의 메모리 접근이 가능
  - 강력하지만 경계/수명 관리 실패 시 오류·보안 문제로 이어질 수 있다.
- Java 레퍼런스(reference): 객체를 가리키는 “참조”이지만 주소 연산은 불가하고, 접근은 JVM의 안전 규칙 안에서 이루어진다.

### 5.2 스택/힙을 “어디에” 두는가

- 네이티브: OS 프로세스의 가상 주소공간 안에 스택/힙이 있으며, 언어/런타임은 그 위에서 동작한다.
- 자바: JVM이 OS 프로세스 위에서 실행되며, JVM의 런타임 데이터 영역(힙/스택 등)을 구성하고 관리한다.
  - 결과적으로 “자바는 스택/힙을 안 쓴다”가 아니라, JVM이 관리하는 방식이 다르다는 것으로 이해하는 것이 정확하다.
