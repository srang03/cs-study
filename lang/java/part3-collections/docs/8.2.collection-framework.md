# Java 컬렉션 프레임워크(Collection Framework) 학습

## 1. 자료구조와 컬렉션

소프트웨어 개발에서 데이터를 효율적으로 저장하고 조회하려면 자료구조가 필요하다. 선형 구조(배열, 연결 리스트)와 비선형 구조(트리, 그래프)로 구분하며, 데이터 양과 접근 패턴에 따라 적절한 구조를 선택해야 한다. 데이터량이 많거나 검색·정렬이 중요할 때는 비선형 구조를 사용하는 것이 성능상 유리한 경우가 많다.

실무에서는 자료구조를 직접 구현하기보다 검증된 라이브러리나 프레임워크를 재사용하는 것이 일반적이다. Java에서는 객체들의 모음을 다루기 위해 컬렉션 프레임워크를 제공한다. DB에서 조회한 소량의 데이터를 애플리케이션 메모리에서 다룰 때도 List, Set 등의 컬렉션에 담아 사용할 수 있으나, 컬렉션 프레임워크 자체는 "메모리 상의 객체 모음"을 위한 API이며 DB/ORM 기술(JDBC, JPA 등)과는 별개의 개념이다.

---

## 2. Java Collection Framework란

Java 컬렉션 프레임워크는 `java.util` 패키지에 포함된 인터페이스와 구현 클래스들의 집합이다. 여러 객체를 하나의 단위로 저장하고, 조회·추가·삭제·순회 등의 연산을 일관된 방식으로 제공한다. 대부분의 자료구조 클래스는 제너릭으로 구현되어 있어 타입 안전하게 사용할 수 있으며, generic.md에서 다룬 제너릭 사용법이 그대로 적용된다.

---

## 3. 주요 인터페이스와 구현체

컬렉션 프레임워크는 크게 컬렉션(원소의 모음)과 맵(키–값 쌍)으로 나뉜다. 인터페이스별 대표 구현체만 정리한다.

### 3.1 List (순서 있음, 중복 허용)

| 구현체 | 내부 구조 | 특징 |
|--------|-----------|------|
| ArrayList | 배열 | 인덱스 접근 빠름, 끝에 추가/삭제에 유리 |
| LinkedList | 연결 리스트 | 앞뒤 삽입·삭제에 유리, 순차 접근 |
| Vector | 배열 | 스레드 안전(동기화), 레거시 |

### 3.2 Set (순서 없음(또는 정렬 순), 중복 불허)

| 구현체 | 내부 구조 | 특징 |
|--------|-----------|------|
| HashSet | 해시 테이블 | 삽입·조회 평균 O(1), 순서 보장 안 함 |
| TreeSet | 트리(레드-블랙) | 정렬 순서 유지, 비교 기반 |

### 3.3 Map (키–값 쌍, 키 중복 불허)

| 구현체 | 내부 구조 | 특징 |
|--------|-----------|------|
| HashMap | 해시 테이블 | 키 기준 삽입·조회 평균 O(1) |
| Hashtable | 해시 테이블 | 스레드 안전(동기화), 레거시 |
| TreeMap | 트리 | 키 정렬 순서 유지 |
| Properties | Hashtable 기반 | 키·값 문자열, 설정 파일 등에 사용 |

## 4. 사용 시 고려사항

- 제너릭 사용: `List<String>`, `Set<Integer>`, `Map<K, V>`처럼 구체 타입을 지정하면 형 안정성과 가독성이 좋아진다.
- 스레드 안전성: 기본 구현체(ArrayList, HashMap, HashSet 등)는 스레드 안전하지 않다. 멀티스레드 환경에서는 동기화된 래퍼(`Collections.synchronizedList` 등), `ConcurrentHashMap` 같은 동시성 전용 구현체, 또는 별도 동기화 전략을 고려해야 한다.
- 선형/비선형 선택: 인덱스 접근·순차 처리 위주면 List(ArrayList/LinkedList), 중복 제거·집합 연산이 필요하면 Set, 키로 값을 찾는 구조면 Map을 선택한다.

### 4.1 ArrayList와 멀티스레드

ArrayList는 스레드 안전하지 않다. 멀티스레드 환경에서 여러 스레드가 동시에 삽입·삭제·조회를 수행하면 동기화 문제(예: `ConcurrentModificationException`, 데이터 불일치)가 발생할 수 있다. 이때 사용할 수 있는 대표적인 방법은 다음과 같다.

1. 동기화 블록/메서드: ArrayList에 접근하는 삽입·삭제·조회 코드를 `synchronized` 블록으로 감싸거나, 해당 작업을 수행하는 메서드에 `synchronized` 키워드를 붙여 한 번에 하나의 스레드만 접근하도록 한다.
2. Vector로 교체: 동기화가 내장된 `Vector`로 교체하면 메서드 단위 동기화가 적용되어 스레드 안전성을 얻을 수 있다. 다만 Vector는 레거시이며, 현대적인 코드에서는 `Collections.synchronizedList(new ArrayList<>())`나 `CopyOnWriteArrayList` 등도 고려할 수 있다.
- Vector는 ArrayList와 같이 배열 기반의 List이지만, 내부적으로 동기화(synchronization) 메커니즘을 포함하고 있어 여러 스레드가 동시에 접근해도 데이터 일관성이 보장됩니다. 스레드 안전이 필요한 상황에 사용할 수 있다.
