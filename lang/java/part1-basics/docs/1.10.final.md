# final과 불변성

## 1. final의 의미

Java에는 C/C++의 `const`처럼 "컴파일 시점 상수"를 나타내는 별도 키워드가 없다.
대신 `final` 키워드로 "최초 한 번만 할당 가능하고, 이후 재할당할 수 없다"는 제약을 둔다.

- `final`이 붙은 변수 또는 필드는 한 번만 할당할 수 있으며, 할당 이후 다른 값(또는 다른 참조)으로 덮어쓰는 것이 컴파일 시점에 금지된다.
- 적용 대상: 인스턴스 필드, 클래스(static) 필드, 지역 변수, 매개변수에 모두 사용할 수 있다.

## 2. 할당 시점과 컴파일 상수

### 2.1 할당 시점

final 변수·필드는 반드시 "선언 시"에 초기화할 필요는 없다.
할당이 일어날 수 있는 시점은 다음과 같다.

- 선언 시: `final int x = 1;` 처럼 선언과 동시에 초기화.
- 인스턴스 필드: 생성자 또는 인스턴스 초기화 블록에서 한 번 할당.
- 클래스(static) 필드: static 초기화 블록에서 한 번 할당.
- 지역 변수: 선언 시 또는 사용 전에 한 번만 할당.

위와 같이 한 번 할당된 이후에는 재할당이 불가능하다.

### 2.2 컴파일 타임 상수

`final` 중에서도 컴파일러가 "상수"로 취급해 값을 인라인할 수 있는 경우가 있다.
이 문서에서 "컴파일 시점" 또는 "컴파일 타임"은 자바 컴파일러가 소스를 바이트코드로 번역하는 시점, 즉 빌드 타임을 의미한다.

- 조건: `static final`이고, 타입이 원시 타입(primitive) 또는 `String`이며, 초기화 식이 상수 표현식(constant expression)인 경우.
- 상수 표현식: 리터럴, 또는 빌드 타임에 값이 정해진 다른 `static final` 상수만으로 구성된 식처럼, 컴파일 시점에 값이 결정되는 표현을 말한다.

이 조건을 만족하면 컴파일러가 해당 필드를 컴파일 타임 상수로 간주하고, 사용처에 값을 직접 넣는 등 최적화할 수 있다.
조건을 만족하지 않는 final(예: 초기값이 메서드 호출 결과이거나 런타임에 결정되는 경우)은 "한 번만 할당되는 변수"이지만, 컴파일 타임 상수는 아니다.

## 3. final이 필요한 이유

### 3.1 재할당 방지

final을 쓰는 주된 목적은 의도치 않은 재할당을 막는 것이다.
프로그램 오류 중 상당수는 "변수나 필드를 잘못된 시점에 다른 값으로 덮어써서" 발생한다.
final을 붙이면 해당 변수·필드에 대한 재할당을 클래스 내부와 외부 모두에서 막을 수 있어, 이런 종류의 실수를 줄일 수 있다.

### 3.2 성능

JVM은 final 필드 등을 활용해 최적화(예: 인라인, 가시성 관련 최적화)를 할 수 있다.
다만 final을 사용하는 1차 목적은 성능이 아니라 재할당 방지와 의도 표현이다.
성능 이점은 부수적인 것으로 보는 것이 맞다.

### 3.3 private setter와의 차이

setter를 private으로 두면 외부에서만 그 필드를 통한 변경 접근이 차단된다.
클래스 내부에서는 여전히 setter나 직접 대입으로 필드를 바꿀 수 있다.

final은 "그 변수·필드에 다른 값을 다시 대입하는 것(재할당)"을 내부와 외부 모두에서 막는다.
즉, "이 변수는 항상 이 값(또는 이 참조)만 가진다"를 언어 수준에서 강제하는 것이다.
객체가 가리키는 내부 상태(필드 값, 컬렉션 요소 등)까지 바꾸지 못하게 하려면, 해당 클래스를 불변으로 설계하는 등 별도 설계가 필요하다.

### 3.4 람다와 익명 클래스 캡처

람다와 익명 클래스는 바깥 메서드의 지역 변수를 "캡처"해 사용할 수 있다. 캡처는 람다·익명 클래스가 생성될 때 그 시점의 변수 값을 복사해 두는 방식으로 이루어진다.

Java는 람다·익명 클래스에서 사용하는 지역 변수가 재할당되면 안 된다고 요구한다(effectively final). 캡처된 뒤 그 지역 변수가 재할당되면, 람다 입장에서는 "캡처 시점의 값"과 "현재 값"이 달라져 어떤 값을 기준으로 동작해야 할지 애매해진다. 또한 람다·익명 클래스는 별도 스레드에서 나중에 실행될 수 있는데, 그때 바깥 변수가 이미 바뀌어 있으면 예측하기 어려운 동작이 된다. 그래서 Java는 "캡처되는 변수는 재할당되지 않는다"를 전제로 두고, 이 요구를 만족하는 변수만 캡처할 수 있게 한다.

`final`을 붙이거나, 재할당하지 않아 effectively final이 되면 이 요구를 만족한다. effectively final이란 `final` 키워드가 없어도 초기화 이후 재할당이 없어 사실상 final과 동일하게 동작하는 상태를 말한다. 이 제약은 원시 타입과 참조 타입 모두에 적용된다. 참조 타입의 경우 "캡처된 참조가 바뀌지 않는다"는 전제가 만족되어야 하며, 참조가 가리키는 객체의 내부 상태 변경은 여전히 허용된다.

## 4. 원시 타입과 참조 타입

### 4.1 원시 타입

`final int x = 1;`처럼 원시 타입에 final을 붙이면, 그 변수가 담는 값 자체를 다시 대입할 수 없다.

### 4.2 참조 타입

참조 타입에 final을 붙이면, 재할당이 불가능한 것은 참조(가리키는 대상)이다.

- 금지되는 것: `final List<String> list = new ArrayList<>();` 선언 후 `list = new ArrayList<>();`처럼 다른 인스턴스를 가리키도록 참조를 바꾸는 것.
- 허용되는 것: `list.add("a");`처럼 같은 인스턴스의 내부 상태(요소, 필드 등)를 변경하는 것은 final이 막지 않는다.

즉, final은 "이 변수가 가리키는 객체가 바뀌지 않는다"만 보장하며, 객체 내부가 변경 가능한지는 해당 클래스 설계에 따른다.

## 5. final 참조가 필요한 이유

참조만 고정되고 객체 내부는 바꿀 수 있어도, "이 변수는 항상 이 인스턴스만 가리킨다"는 제약만으로도 다음 이점이 있다.

- 의도 표현과 실수 방지: 다른 객체로 참조를 바꾸는 재할당을 막아, "이 변수가 가리키는 대상은 하나"라는 불변식을 코드로 명시할 수 있다.
- 동시성: Java 메모리 모델(JMM)에 따르면, 생성자가 끝난 뒤에 다른 스레드가 그 객체를 볼 때, final 필드는 별도 동기화 없이도 생성자에서 쓴 값이 보인다(safe publication). 따라서 "참조가 바뀌지 않는 final 필드"는 스레드 간 객체 공유 시 초기화 가시성 보장에 의미가 있다.
- 람다·익명 클래스 캡처: 바깥 메서드의 지역 변수를 캡처할 때 effectively final이어야 한다. 상세는 3.4를 참고한다.
