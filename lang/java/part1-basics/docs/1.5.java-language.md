# 자바 언어

## 1. Java 특징

JVM 기반에서 작동하는 OOP 언어이다.
C/C++에서 개발자가 직접 다루는 메모리 관리와 그에 따른 책임 이슈를 구조적으로 제거한 managed 코드이다.
OS와 플랫폼에 의존성이 없고 이식성이 높다.

## 2. JVM

JVM은 애플리케이션 레이어에서 동작하는 유저 모드 애플리케이션이므로 OS에 영향을 미칠 수 없다.
JVM은 빅엔디안 체계를 사용한다.

- 엔디안: 다중 바이트를 메모리에 저장하는 순서이다. 빅엔디안은 상위(높은) 바이트가 낮은 메모리 주소에 오는 방식이고, 리틀엔디안은 상위 바이트가 높은 메모리 주소에 오는 방식이다.
- 네이티브 언어에서 비트 연산을 다룰 때는 플랫폼의 엔디안 체계를 고려해야 한다.

가상 메모리 관리 체계도 JVM에 포함되어 있다.
C/C++은 메모리 접근에 대한 극단적인 자유도를 가지게 되고, 이에 따른 관리와 책임은 개발자가 수행한다. 메모리 관리에서는 "반환"과 "경계"가 중요하다. 반환을 하지 않으면 누수가 발생하고, 경계를 벗어나면 잘못된 접근·연산 오류 및 OS 보안에 영향을 줄 수 있다.
Java는 메모리 할당은 수행하지만 반환 개념은 없다. GC가 회수한다.

## 3. JVM 구성 요소

### 3.1 ClassLoader (초기 설정)

실행을 위해 클래스를 로딩한다.

- Loading: Bootstrap / Extension / Application class loader
- Linking: Verify → Prepare → Resolve
- Initialization

### 3.2 Runtime data area (메모리)

- Method area
  - Runtime constant pool
- Heap
  - 객체를 인스턴스화할 때 사용한다.
  - GC를 통한 메모리 자동 회수의 대상이다.
- Stack
  - 스레드마다 고유한 영역을 가진다.
- PC register
  - 스레드마다 다음에 실행할 명령의 주소를 가리킨다.
- Native method stack

### 3.3 Execution engine (JVM 기계어 실행)

바이트코드를 실제로 실행한다.

- Interpreter: 바이트코드를 CPU가 인식할 수 있는 기계어로 번역한다.
- JIT compiler: 자주 실행되는 바이트코드를 기계어로 컴파일해 두었다가 재사용한다.
- GC

## 4. JIT(Just-In-Time) 컴파일러

JIT 컴파일러는 자바 바이트코드를 실제 기계어로 번역한다.
실행 기록(프로파일링)을 바탕으로 자주 실행되는 구간(hot spot)을 파악하고, 그 코드를 기계어로 컴파일해 두었다가 재사용한다. 반복문 등에 주로 적용되며, 프로그램을 오래 실행할수록 성능 개선에 유리하다.

## 5. 자바의 장점

애플리케이션 개발·서비스 제공과 OS·H/W 이해를 별개의 문제로 분리한다. 즉 애플리케이션 개발에 집중하기 위한 언어이다.
최적화, 메모리 관리 등 시스템 수준에서 이루어져야 할 일을 JVM이 대신 수행하여, 개발자에게 더 높은 추상성을 주고 단순화가 가능해지며, 그에 따라 유지보수성이 높아진다.
