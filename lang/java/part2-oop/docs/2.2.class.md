# 클래스

## 1. 객체와 클래스

객체는 OOP에서 소프트웨어를 구성하는 단위이다. 필드와 메서드가 모여 하나의 집합체를 이룬다.
Java에서는 클래스로 객체를 정의한다. 클래스는 존재 이유와 목적이 명확해야 재사용과 유지보수가 쉬워진다.

## 2. 코드 구성 단위의 계층

피연산자가 모여 식(expression)이 되고, 식이 모여 구문(statement)이 된다. 구문이 모여 메서드, 메서드가 모여 클래스, 클래스가 모여 패키지를 이룬다. 이 계층을 이해하면 클래스가 전체 구조에서 차지하는 위치를 파악할 수 있다.

## 3. 클래스, 객체, 인스턴스

클래스는 객체를 기술하는 문법이자 자료형이다.

- 클래스 타입 변수는 참조형이며 `null`을 가질 수 있다.
- 인스턴스는 클래스로부터 생성된 객체이다. 같은 클래스에서 나온 객체라도 각각 별도의 인스턴스이다.

## 4. 클래스 기본 문법

관계(상속 등)를 생각하지 않으면, 클래스는 필드와 메서드로 이루어진 단순 집합체로 볼 수 있다.
구성 요소를 멤버(member)라 부르며, 변수는 필드(Field), 함수는 메서드(Method)이다. 변수는 메모리를 사용하는 기본 수단이고, 클래스는 선언과 정의가 한 곳에 함께 있는 구조이다. 클래스를 설계할 때는 객체를 만드는 쪽과 사용하는 쪽을 구분해서 생각하는 것이 중요하다.

## 5. 클래스 기본 구성

구성 단위는 패키지 > 클래스 > 생성자, 메서드, 필드 순이다.

- 클래스 이름은 영문으로 작성하고 첫 글자는 대문자로 쓰는 것이 관례이다.
- 클래스 이름과 `.java` 파일명이 같아야 한다. `public` 클래스일 때는 반드시 지켜야 하며, `public` 클래스는 파일당 하나만 둘 수 있다.

## 6. 멤버 선언 및 초기화

필드 선언 시 초깃값을 줄 수 있다. 초깃값을 주지 않으면 원시 타입은 0, 참조형은 `null`로 초기화된다.
생성자에서도 초기화할 수 있으며, 필드 초깃값과 생성자 초기화가 모두 있으면 생성자에서 설정한 값이 최종적으로 적용된다. 이렇게 두 경로를 두는 이유는 선언 시 기본값을 두고, 생성자에서 매개변수 등으로 구체적인 값을 넣을 수 있게 하기 위해서이다.

## 7. 생성자

생성자는 객체가 생성될 때 자동으로 호출되는 특수한 메서드이다.

- 이름은 클래스 이름과 같다.
- 반환 타입을 적지 않으며, `new` 연산자와 함께만 호출된다. 일반 메서드처럼 `obj.생성자()` 형태로는 호출할 수 없다.
- 생성자를 하나도 정의하지 않으면 컴파일러가 인자 없는 빈 생성자를 자동으로 추가한다.
- 오버로딩이 가능하므로, 매개변수 목록이 다른 여러 생성자를 둘 수 있다.
- 생성자에는 클래스(객체) 초기화에 필요한 코드만 두고, 다른 책임은 메서드로 분리하는 것이 좋다.

## 8. 인스턴스 동적 생성

`new` 연산자로 인스턴스를 동적으로 생성한다.

- 클래스 타입 변수는 모두 참조형이므로, 인스턴스가 없을 때는 `null`을 가진다.
- 하나의 인스턴스를 여러 참조가 가리킬 수 있다. 이때 한 참조를 통해 상태를 바꾸면 다른 참조를 통해서도 같은 변경이 보이므로, 의도치 않은 부수 효과(side effect)를 고려해야 한다.

## 9. 접근 제어자

접근 제어자는 멤버에 대한 접근 범위를 제한한다. 내부 구현을 숨기고, 외부에서 의도치 않게 변경하는 것을 막기 위함이다.

- `public`: 모든 패키지에서 접근 가능하다.
- `protected`: 같은 패키지와 파생(하위) 클래스에서만 접근 가능하다.
- `default`(패키지-private): 접근 제어자를 쓰지 않으면 적용되며, 같은 패키지 내에서만 접근 가능하다.
- `private`: 해당 클래스 내부에서만 접근 가능하다.

## 10. getter와 setter

`private` 필드는 클래스 밖에서 직접 접근할 수 없다. 외부에서 읽거나 쓸 수 있게 하려면 getter·setter 메서드를 제공한다.

- getter: 필드 값을 반환한다.
- setter: 필드 값을 설정한다.

읽기와 쓰기를 메서드로 분리하면, 검증·로깅·변경 알림 등을 넣어 접근을 제어할 수 있다. IDE에서 getter·setter 자동 생성 기능을 지원한다.
객체지향에서는 객체를 설계·구현하는 관점과, 그 객체를 사용하는 관점을 분리해서 생각하는 것이 중요하다.

## 11. this

`this`는 현재 인스턴스(객체 자신)를 가리키는 참조이다.

- 인스턴스 메서드 안에서만 사용할 수 있으며, `static` 메서드에서는 사용할 수 없다. `static` 메서드는 클래스에 속하고 인스턴스에 종속되지 않기 때문이다.
- 같은 클래스의 서로 다른 인스턴스는 각각 다른 `this` 값을 가진다. `this`는 "이 메서드를 호출한 그 객체"를 의미한다.
- 메서드 호출 시 `this` 참조는 스택 프레임에 저장되어, 인스턴스 필드 접근 등에 사용된다.

## 12. 클래스 설계 시 주의사항

### 12.1 단일 책임 원칙(SRP)

클래스는 하나의 변경 이유만 가져야 한다. 책임이 여러 개면 수정 시 영향 범위가 넓어지고, 테스트와 유지보수가 어려워진다. 하나의 클래스가 너무 많은 일을 하면 응집도가 낮아지고, 다른 클래스와의 결합도가 높아진다. 책임을 분리하면 각 클래스가 명확한 역할을 갖고, 변경 시 해당 클래스만 수정하면 된다.

### 12.2 상태 변경과 불변성

가변(mutable) 객체는 여러 참조가 공유할 때 예측하기 어려운 부수 효과를 일으킬 수 있다. 가능하면 불변(immutable)으로 설계하거나, 변경 가능한 범위를 최소화한다. 불변 객체는 생성 후 상태가 바뀌지 않으므로 스레드 안전하고, 캐싱·공유가 안전하다. 필드가 많거나 생성 비용이 크면 Builder 패턴으로 생성 과정을 분리하는 것이 좋다.

### 12.3 캡슐화와 정보 은닉

필드는 `private`으로 두고, 필요한 경우에만 getter·setter로 노출한다. setter를 무분별히 두면 객체의 불변식(invariant)이 깨질 수 있다. 내부 구현이 바뀌어도 외부 인터페이스가 유지되도록 설계하면, 리팩터링과 확장이 쉬워진다.

### 12.4 의존성 방향과 결합도

클래스는 구체보다 추상(인터페이스, 추상 클래스)에 의존하는 것이 좋다. 구체 클래스에 직접 의존하면 교체와 테스트가 어렵다. 의존성 주입(DI)을 사용하면 생성 시점에 의존 객체를 주입받아 결합도를 낮출 수 있다.

### 12.5 디자인 패턴 관점

- Factory / Factory Method: 객체 생성 로직을 별도 메서드나 클래스로 분리하여, 생성 방식 변경 시 한 곳만 수정한다.
- Builder: 필드가 많거나 선택적 초기화가 필요할 때 생성자를 여러 개 두는 대신, 가독성 있는 단계별 생성이 가능하다.
- Strategy / Template Method: 행위를 교체 가능하게 두어, 클래스 수정 없이 확장할 수 있게 한다.

## 13. 생성자 설계 시 주의사항

### 13.1 생성자에서의 로직 최소화

생성자는 객체를 유효한 상태로 만드는 데 필요한 작업만 수행한다. 비즈니스 로직, I/O, 네트워크 호출, 무거운 계산 등은 생성자에서 피하고, 필요 시 지연 초기화나 별도 메서드로 분리한다. 생성자가 무거우면 객체 생성 비용이 커지고, 테스트 시 모킹이 어려워진다.

### 13.2 생성자 오버로딩과 중복

여러 생성자가 비슷한 초기화를 할 때, 한 생성자에서 다른 생성자를 `this()`로 호출하여 중복을 줄인다. 공통 로직이 한 곳에 모이면 수정 시 일관성이 유지된다.

### 13.3 정적 팩토리 메서드

`new` 대신 `static` 메서드로 객체를 반환하는 방식이다. 생성자만으로는 표현하기 어려운 "의미 있는 생성 방식"을 메서드 이름으로 드러낼 수 있다(예: `List.of()`, `Optional.ofNullable()`). 캐싱, 하위 타입 반환 등 생성자로는 표현하기 어려운 동작을 담을 수 있다.

### 13.4 생성 실패와 예외

생성자에서 매개변수 검증이 실패하면 `IllegalArgumentException` 등을 던진다. 생성자가 완료되기 전에 예외가 나면 객체는 생성되지 않으며, 호출자에게 참조가 반환되지 않는다. `final` 필드는 생성자 완료 전에만 설정 가능하므로, 예외 시 일부만 초기화된 불완전한 객체가 외부에 노출되지 않는다.

### 13.5 생성자에서 가상 메서드 호출 금지

생성자에서 오버라이드 가능한 메서드를 호출하면, 하위 클래스의 메서드가 아직 초기화되지 않은 상태에서 실행될 수 있다. 따라서 생성자에서는 오버라이드 가능한 메서드를 호출하지 않는 것이 안전하다.
