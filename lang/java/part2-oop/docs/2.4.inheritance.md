# 상속

## 1. 상속 기본

상속은 `extends` 키워드로 부모 클래스의 멤버를 자식 클래스에서 재사용(확장)하는 문법이다.

```java
class Child extends Parent { }
```

- 부모/자식, 기본/파생 클래스로 구분한다. is-a 관계(자식은 부모의 한 종류)에 적합하다. has-a 관계(구성)는 필드로 객체를 갖는 방식이 더 적합한 경우가 많다.
- Java는 다중 상속을 허용하지 않는다. 한 클래스는 하나의 클래스만 상속할 수 있어, 다이아몬드 문제 등을 피한다.
- 모든 클래스는 `Object`를 최상위 조상으로 가진다. `Object`를 명시하지 않아도 자동으로 상속된다.

## 2. 상속에서의 멤버 접근과 메모리 구조

파생 클래스 인스턴스는 부모 클래스의 멤버를 포함한 하나의 객체이다. 부모 멤버와 자식 멤버가 한 인스턴스 안에 함께 있다.

- `private`: 파생 클래스에서도 접근할 수 없다. 부모가 캡슐화한 내부 구현이다.
- `protected`: 같은 패키지와 파생 클래스에서 접근 가능하다. 상속 관계에서 하위 클래스가 부모 멤버를 사용할 수 있게 하기 위함이다.

파생 클래스에서 부모의 `private` 필드를 그대로 선언하는 것은 부모와 중복 정의가 되어 적절하지 않다. 부모가 이미 가진 멤버를 다시 정의하면 혼란과 오류를 일으킬 수 있다.

## 3. 상속과 생성자 호출 순서

파생 클래스 생성자가 호출되면, 그 본문이 실행되기 전에 부모 생성자가 먼저 실행된다. `super()`를 명시하지 않으면 컴파일러가 인자 없는 `super()`를 생성자 첫 줄에 넣는다.

호출 순서는 재귀와 비슷하게 스택에 쌓인다. `new Child()` 시:

1. `Child()` 진입 → `super()` 호출로 `Parent()`로 이동
2. `Parent()` 실행 완료
3. `Child()` 본문 실행

이렇게 부모가 먼저 초기화된 뒤 자식이 초기화되므로, 자식 생성자에서 부모 필드를 사용할 때는 이미 유효한 상태가 보장된다.

## 4. super()로 부모 생성자 선택

파생 클래스 생성자에서 부모 클래스의 여러 생성자 중 하나를 명시적으로 호출하려면 `super(인자)`를 사용한다. `super()`는 생성자 본문의 첫 문장이어야 한다.

```java
class Parent {
    Parent() { }
    Parent(int x) { }
}
class Child extends Parent {
    Child() {
        super();      // 부모 기본 생성자
    }
    Child(int x) {
        super(x);     // 부모 생성자 중 하나 선택
    }
}
```

## 5. 생성자에서 부모 필드 덮어쓰기 주의

자식 생성자에서 부모가 초기화한 필드를 `null` 등으로 덮어쓰면, 부모 메서드가 잘못된 상태에서 동작한다. 아래 예에서 `TestEx` 생성자가 `string`을 `null`로 덮어쓰면, `getString()` 호출 시 `NullPointerException`이 발생한다.

```java
class Test {
    protected byte[] string;
    Test() {
        string = new byte[]{'h', 'e', 'l', 'l', 'o'};
    }
    String getString() {
        return new String(string);  // string이 null이면 NPE
    }
}
class TestEx extends Test {
    TestEx() {
        super();
        string = null;  // 부모가 초기화한 필드를 덮어씀 → getString() 시 NPE
    }
}
```

자식 생성자에서는 부모가 초기화한 필드를 함부로 바꾸지 않고, 필요한 경우 부모 생성자에 넘길 인자를 조정하는 방식이 안전하다.

## 6. 메서드 재정의(Overriding)

파생 클래스에서 부모 메서드와 같은 시그니처(이름, 매개변수 목록)로 메서드를 다시 정의하면 재정의(override)가 된다. 오버로딩(다중 정의)은 매개변수 목록이 다른 경우이므로 구분해야 한다.

재정의된 메서드는 기존 동작을 대체하거나, `super.메서드()` 호출 후 추가 로직을 넣는 식으로 확장할 수 있다. 메서드 호출 시에는 변수의 선언 타입이 아니라 실제 인스턴스의 타입에 정의된 메서드가 실행된다(동적 바인딩).

```java
Test t1 = new TestEx();   // t1은 Test 타입 변수, 실제 인스턴스는 TestEx
TestEx t2 = new TestEx(); // t2는 TestEx 타입 변수
// t1과 t2 모두 TestEx에 정의된 재정의 메서드가 호출됨
```

## 7. @Override 어노테이션

`@Override`를 재정의하는 메서드 위에 붙이면, 컴파일러가 부모에 같은 시그니처의 메서드가 있는지 검사한다. 오타나 시그니처 불일치로 인한 실수를 줄이기 위함이다.

```java
@Override
public String toString() {
    return "Child";
}
```

## 8. final로 재정의 차단

부모 메서드에 `final`을 붙이면 파생 클래스에서 재정의할 수 없다. 설계상 "이 메서드는 변경되면 안 된다"는 의도를 강제할 때 사용한다.

```java
public final void getData() { }
```

## 9. 템플릿 메서드 패턴과 훅 메서드

부모가 정한 흐름 안에서, 파생 클래스가 특정 단계만 바꿀 수 있게 하는 패턴이다. 부모의 `public` 메서드가 내부에서 `protected` 메서드(훅)를 호출하고, 파생 클래스는 그 훅만 재정의한다. 프레임워크가 "틀"을 제공하고, 사용자가 "일부"만 채우는 방식이다.

```java
// 부모: setData()가 전체 흐름을 정하고, onSetData()만 자식이 바꿀 수 있게 함
class DataHolder {
    protected int data;

    public void setData(int param) {
        if (onSetData(param)) {   // 훅 호출 → 자식이 재정의하면 자식 버전 실행
            this.data = param;
        } else {
            this.data = 0;
        }
    }

    protected boolean onSetData(int param) {
        return true;  // 기본: 검증 없이 항상 허용
    }

    public int getData() {
        return data;
    }
}

// 자식: onSetData()만 재정의해 0~100 범위 검증 추가
class BoundedDataHolder extends DataHolder {
    @Override
    protected boolean onSetData(int param) {
        return param >= 0 && param <= 100;
    }
}

// 사용
BoundedDataHolder holder = new BoundedDataHolder();
holder.setData(50);   // onSetData(50) → true → data = 50
holder.setData(-5);   // onSetData(-5) → false → data = 0
holder.setData(200);  // onSetData(200) → false → data = 0
System.out.println(holder.getData());  // 0 (마지막 setData(200)이 거부됨)
```

`setData()`는 부모가 정의한 "검증 후 저장" 흐름을 그대로 두고, `onSetData()`만 자식에서 재정의해 검증 규칙을 바꾼다. 이렇게 하면 부모 구조를 유지하면서 자식이 필요한 부분만 확장할 수 있다.

## 10. 상속 설계 시 주의사항

### 10.1 상속 깊이

상속이 여러 단계로 이어지면 코드 흐름을 따라가기 어렵다. 가능하면 상속 단계를 줄이고, 구성(composition)을 고려한다.

### 10.2 미래 확장

파생 클래스는 부모보다 나중에 정의될 수 있다. 부모를 설계할 때 "파생 클래스가 이 메서드를 재정의할 수 있게 해야 하는가"를 고려하고, `protected` 훅 메서드 등으로 확장 지점을 두는 것이 좋다.

### 10.3 생성자에서 가상 메서드 호출 금지

부모 생성자에서 재정의 가능한 메서드를 호출하면, 그 시점에 자식 생성자는 아직 실행되지 않았다. 그런데 동적 바인딩 때문에 부모가 호출한 메서드가 자식에서 재정의된 버전으로 실행된다. 자식 필드는 아직 기본값(0, null)이므로, 자식 메서드가 그 필드를 사용하면 잘못된 동작이나 `NullPointerException`이 발생한다.

```java
class Parent {
    Parent() {
        printInfo();  // 위험: 재정의 가능한 메서드 호출
    }
    void printInfo() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    private String name = "Child";  // 생성자 실행 전에는 기본값 null

    Child() {
        super();  // 여기서 Parent() → printInfo() 호출
        name = "Child";  // 이 시점에야 name 초기화
    }

    @Override
    void printInfo() {
        System.out.println(name.length());  // name이 아직 null → NPE
    }
}

// new Child() 호출 시:
// 1. Child() 진입 → super() → Parent() 실행
// 2. Parent() 안에서 printInfo() 호출
// 3. 실제 인스턴스가 Child이므로 Child.printInfo() 실행 (동적 바인딩)
// 4. Child의 name은 아직 초기화 전 → null → name.length()에서 NullPointerException
```

부모 생성자가 `printInfo()`를 호출할 때, 자식의 `printInfo()`가 실행되지만 자식 생성자 본문은 아직 실행되지 않아 `name`이 `null`이다. 따라서 생성자에서는 재정의 가능한 메서드를 호출하지 않는 것이 안전하다. `final` 메서드나 `private` 메서드는 재정의되지 않으므로 호출해도 된다.
