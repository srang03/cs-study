# 객체 관계

## 1. 클래스 간 관계 개요

클래스 설계는 목표를 이루기 위한 객체와 그 관계를 정하는 것이다. 관계는 보통 UML(Unified Modeling Language)로 표현한다.

주요 관계 유형은 다음과 같다.

- 일반화(Generalization): 상속 관계. 자식은 부모의 한 종류(is-a)이다.
- 실체화(Realization): 인터페이스 구현. 클래스가 인터페이스의 계약을 구현한다.
- 합성(Composition): 강한 소유 관계. 포함된 객체의 생명주기가 소유자와 함께한다.
- 집합(Aggregation): 약한 포함 관계. 포함된 객체가 독립적으로 존재할 수 있다.
- 연관(Association): 한 클래스가 다른 클래스를 필드로 갖는 관계. 참조로 연결된다.
- 의존(Dependency): 한 클래스가 다른 클래스를 메서드 매개변수·반환값·지역 변수로만 사용하는 관계. 필드로 보유하지 않는다.

## 2. 합성(Composition)과 집합(Aggregation)

클래스는 다른 클래스를 필드로 가질 수 있다. 이때 소유 강도에 따라 합성과 집합으로 구분한다.

### 2.1 합성(Composition)

포함된 객체가 소유자 없이 독립적으로 의미가 없을 때 사용한다. 소유자가 생성될 때 함께 생성되고, 소유자가 사라질 때 함께 사라진다.

```java
class Engine { }
class Car {
    private Engine engine = new Engine();  // Car가 Engine을 소유, Car 생성 시 함께 생성
}
```

`Car`가 `Engine`을 직접 생성해 갖는다. `Car`가 GC 대상이 되면 `Engine`도 함께 회수된다. "부분이 전체에 종속된다"는 관계에 적합하다.

### 2.2 집합(Aggregation)

포함된 객체가 외부에서 생성되어 전달되거나, 여러 소유자가 공유할 수 있을 때 사용한다. 생명주기가 소유자와 분리된다.

```java
class Department { }
class Company {
    private List<Department> departments;  // 외부에서 주입되거나, 여러 Company가 공유할 수 있음
    public Company(List<Department> departments) {
        this.departments = departments;
    }
}
```

`Department`는 `Company` 밖에서 생성될 수 있고, `Company`가 사라져도 `Department`는 남을 수 있다. "모임" 관계에 적합하다.

## 3. 의존(Dependency)과 연관(Association)

### 3.1 연관(Association)

한 클래스가 다른 클래스를 필드로 갖는 관계이다. 참조를 통해 지속적으로 연결된다.

```java
class UserData {
    private String name;
    private String phone;
    // getter, setter
}
class Order {
    private UserData customer;  // Order가 UserData를 필드로 보유 → 연관
}
```

### 3.2 의존(Dependency)

한 클래스가 다른 클래스를 필드가 아닌 매개변수·반환값·지역 변수로만 사용하는 관계이다. 특정 기능 수행 시에만 필요하다.

```java
class ReportService {
    public String format(UserData user) {  // UserData를 매개변수로만 사용 → 의존
        return user.getName() + " - " + user.getPhone();
    }
}
```

의존은 연관보다 결합도가 낮다. `ReportService`는 `UserData`를 필드로 갖지 않아, `UserData` 변경 시 영향 범위가 작다.

## 4. UML과 다이어그램

UML은 클래스·객체·연관 관계를 시각화하는 표준 모델링 언어이다.

- 클래스 다이어그램: 클래스, 필드, 메서드, 관계(상속, 연관, 의존 등)를 표현한다. 설계 단계에서 구조를 파악하는 데 사용한다.
- 시퀀스 다이어그램: 객체 간 메시지 호출 순서를 시간 흐름에 따라 표현한다. 특정 시나리오의 동작을 분석할 때 사용한다.

클래스 다이어그램과 시퀀스 다이어그램은 설계·리뷰·문서화에서 자주 쓰이므로 익혀 두는 것이 좋다.

## 5. 동적 바인딩

메서드 호출 시 "어떤 구현을 실행할지"를 결정하는 것을 바인딩이라 한다.

- 정적 바인딩(Static binding): 컴파일 시점에 결정된다. `private`, `static`, `final` 메서드, 오버로딩된 메서드 등이 해당한다.
- 동적 바인딩(Dynamic binding): 실행 시점에 실제 인스턴스 타입에 따라 결정된다. 재정의(override)된 메서드가 해당한다. Late binding이라고도 한다.

Java에서는 재정의된 메서드를 호출할 때, 변수의 선언 타입이 아니라 실제 인스턴스의 타입에 정의된 메서드가 실행된다. JVM이 인스턴스 타입을 보고 적절한 메서드를 찾아 호출한다. 이 때문에 다형성(polymorphism)이 동작한다.

## 6. 정리
- 의존: B를 A의 필드로 갖지 않고, 메서드의 매개변수·반환 타입으로만 사용한다.
- 합성(연관): A가 B를 직접 생성해 필드로 갖고, B의 생명주기는 A에 종속된다.
- 집합(연관): A가 B를 외부에서 주입받아 필드로 갖고, B와 A의 생명주기는 독립적이다.
