# 1.2 입출력 — 심화 정리

> 1.2.io.md 보조 문서. 문자열·버퍼·입출력의 CS 이론과 동작 원리를 논리적으로 정리한다.

---

## 1. 핵심 원리

OS 호출 횟수 감소가 성능 차이의 핵심이다.

| 방식 | OS 호출 | 동작 |
|------|---------|------|
| 버퍼 없음 (InputStream 직접) | 매번 | 바이트 1개 읽을 때마다 OS에 요청 |
| 버퍼 있음 (BufferedReader) | 가끔 | 한 번에 많이 읽어 내부 버퍼에 저장 → 이후에는 그 버퍼에서만 꺼냄 |

- 입력: OS 버퍼에서 하나씩 꺼내느냐 vs 한 번에 많이 꺼내서 임시 버퍼에 두고 쓰느냐
- 출력: 매번 flush하느냐 vs 버퍼에 모았다가 한 번에 flush하느냐

---

## 2. 입력 계층 구조

```
[키보드] → [OS 커널 버퍼] → [System.in (InputStream)] → [InputStreamReader] → [BufferedReader 내부 버퍼]
```

### 2.1 버퍼의 계층 (서로 다름)

| 구분 | 위치 | 역할 |
|------|------|------|
| OS 커널 버퍼 | 커널 공간 | 터미널 입력을 Enter까지 모아두고, 프로세스 read() 시 전달 |
| BufferedReader 버퍼 | JVM 힙(사용자 공간) | InputStreamReader에서 읽은 문자를 모아두고, readLine() 등이 이 버퍼에서 반환 |

둘 다 "데이터를 모아두는 곳"이지만 계층과 목적이 다르다.

---

## 3. 문자열(String) 기초

- 문자: 사람이 읽는 단위 (예: 'A', '가', '1')
- 바이트: 컴퓨터 저장·전송 단위 (8비트)
- 인코딩: 문자 ↔ 바이트 변환 규칙 (UTF-8, ASCII 등)

예: `"10 20"` → UTF-8로 `31 30 20 32 30` 같은 바이트 시퀀스로 저장·전송됨.

---

## 4. InputStream

- `InputStream.read()`: 바이트 1개를 읽는다. (Java API: "Reads the next byte of data from the input stream")
- `InputStream` 자체는 버퍼를 갖지 않는다.
- `read(byte[] b)`의 기본 구현은 내부적으로 `read()`를 반복 호출한다.
- `System.in`은 `InputStream`이므로, `read()` 호출마다 OS `read` 호출이 발생한다.

---

## 5. BufferedReader

### 5.1 역할

OS에서 한 번에 많이 읽어와 내부 임시 버퍼(char[])에 캐싱한다. 이후 `readLine()` 등은 이 버퍼에서만 꺼낸다.

### 5.2 버퍼 크기

- 지정 가능: `BufferedReader(Reader in, int sz)` 생성자
- 기본값: 8192 문자 (약 8KB)
- 버퍼 크기는 바이트/문자 개수로 정해지며, 공백·개행으로 "자르는" 것이 아니다.

### 5.3 readLine() 동작

- 버퍼가 비면 `InputStreamReader.read()`를 호출해 버퍼 크기만큼 채운다.
- `readLine()`은 버퍼에서 개행(`\n`, `\r`, `\r\n`)을 만날 때까지 읽어 한 줄을 반환한다.
- 한 줄이 버퍼보다 길면, 버퍼를 여러 번 채우며 `\n`을 찾을 때까지 읽는다.

### 5.4 readLine() vs Scanner.nextLine()

반환 결과는 같다. 둘 다 한 줄 문자열을 반환한다.

차이는 데이터를 가져오는 방식이다.

| | BufferedReader | Scanner |
|---|---------------|---------|
| 내부 버퍼 | 있음 (기본 8192자) | 있음 (더 작음) |
| 읽기 방식 | 한 번에 많이 읽어 버퍼에 채움 | 토큰/라인 단위로 읽음 |
| 파싱 | 없음 (문자열만 반환) | 정규식 기반 파싱 |

BufferedReader는 OS에서 한 번에 더 많이 읽어 내부 버퍼에 두고, `readLine()`은 그 버퍼에서 `\n`까지 꺼내 반환한다.

### 5.5 BufferedReader와 StringTokenizer, Integer.parseInt

BufferedReader 안에 StringTokenizer나 Integer.parseInt가 없다. 별도 클래스·메서드다.

- `BufferedReader`: `readLine()`으로 한 줄 문자열만 반환
- `StringTokenizer`: `java.util` 패키지의 별도 클래스
- `Integer.parseInt()`: `java.lang`의 정적 메서드

코딩 테스트에서는 이 세 가지를 조합해서 사용한다.

## 6. StringTokenizer

### 6.1 기본 구분자

스페이스만이 아니라 공백 문자 전체(whitespace)다.

- 스페이스 `' '`
- 탭 `'\t'`
- 줄바꿈 `'\n'`, `'\r'`
- 폼 피드 `'\f'`

Java 문서: 기본 구분자는 `" \t\n\r\f"`이다.

### 6.2 사용 목적

- 원하는 구분자로 분리하기 위함
- 기본: 공백. 생성자로 지정 가능: `new StringTokenizer(line, ",")`
- `Scanner`는 `nextInt()`, `next()` 등으로 토큰을 바로 읽지만 정규식을 사용해 느리다.
- `StringTokenizer`는 단순 반복문 기반이라 가볍고, 구분자 제어가 명확하다.

### 6.3 Scanner와 조합

`Scanner.nextLine()`으로 읽은 문자열을 `StringTokenizer`에 넘길 수 있다.

```java
Scanner sc = new Scanner(System.in);
String line = sc.nextLine();
StringTokenizer st = new StringTokenizer(line);
int a = Integer.parseInt(st.nextToken());
```

## 7. Scanner vs BufferedReader

### 7.1 입력 소스

둘 다 동일한 System.in을 사용한다. 같은 OS 표준 입력 스트림에서 읽는다.

### 7.2 사용 기준

- 입력량으로 구분하는 것이 맞다. (단일 문자 vs 긴 문자열이 아님)
- 소규모(수십~수백 줄): Scanner도 충분
- 대규모(10만 줄 이상): BufferedReader 권장
- Scanner가 느린 이유는 정규식 파싱 비용이다.

### 7.3 Scanner가 느린 이유

1. 정규식 기반 파싱: `nextInt()`, `nextDouble()` 등이 내부적으로 정규식으로 토큰을 찾음
2. 토큰 단위 처리: 호출마다 정규식 매칭·파싱·변환 수행
3. 파싱 비용이 I/O 비용보다 클 수 있음 (대량 입력 시)

### 7.4 BufferedReader가 빠른 이유

1. 역할 분리: 읽기만 담당, 파싱은 별도
2. 단순 처리: `readLine()` → 문자열 반환, 정규식 없음
3. 대량 입력에 유리: OS 호출 횟수 감소

---

## 8. 출력과 flush

### 8.1 System.out vs BufferedWriter

동일한 버퍼가 아니다. 각각 별도의 내부 버퍼를 가진다.

```
BufferedWriter.write() → [BufferedWriter 내부 버퍼]
                              ↓ flush()
                    OutputStreamWriter
                              ↓
                    [PrintStream (System.out) 내부 버퍼]
                              ↓ flush()
                         OS (실제 출력)
```

### 8.2 System.out (PrintStream)

- 내부 버퍼가 있고 autoFlush가 켜져 있다.
- `println()` 호출 시마다, 또는 `\n`을 쓸 때마다 자동으로 flush된다.
- `println` 반복 → 매번 OS에 쓰기 → 호출 횟수 증가

### 8.3 BufferedWriter

- 자체 내부 버퍼를 가진다.
- `write()`는 이 버퍼에만 쌓고, flush 호출 전까지 OS로 보내지 않는다.
- `flush()` 또는 `close()` 시점에 한 번에 OS로 전달한다.
- 같은 stdout으로 보내되, 더 큰 내부 버퍼로 모아서 한 번에 보낸다.

## 9. 지역성(Spatial Locality)과 캐시

### 9.1 정의

공간 지역성: 한 주소를 접근한 뒤, 인접한 주소를 곧 접근할 가능성이 높다는 특성.

### 9.2 캐시와의 관계

- CPU 캐시는 캐시 라인 단위(보통 64바이트)로 메모리를 가져온다.
- 한 주소를 읽으면 그 주소가 포함된 전체 캐시 라인이 로드된다.
- 이후 인접 주소를 읽으면 이미 캐시에 있으므로 캐시 히트가 된다.

### 9.3 버퍼와의 연결

- BufferedReader 내부 버퍼는 연속된 char[] 배열이다.
- `readLine()`은 이 배열을 순차적으로 스캔해 `\n`을 찾는다.
- 순차 접근 → 공간 지역성 → 캐시 라인 단위로 효율적 사용 → 캐시 히트율이 높다.

---

## 10. 정리: 캐싱 버퍼 객체

BufferedReader와 BufferedWriter는 OS I/O 호출 횟수를 줄이고 스트림 데이터를 효율적으로 처리하기 위한 애플리케이션 레벨 캐싱 버퍼 객체다.

- 입력: OS에서 한 번에 많이 읽어 내부에 보관 → `readLine()` 등은 그 버퍼에서만 꺼냄
- 출력: `write()`로 내부 버퍼에 모았다가 `flush()` 시 한 번에 OS로 전달

| 핵심 | OS 호출 횟수 감소 — 한 번에 많이 vs 하나씩 |
| 문자열 | 문자들의 나열, 바이트로 인코딩되어 저장·전송 |
| 버퍼 | 데이터를 모아두는 메모리, I/O 호출 횟수 감소 |

| InputStream | 버퍼 없음, read()는 바이트 1개 |
| BufferedReader | 내부 버퍼(기본 8192자), 지정 가능 |
| StringTokenizer | 기본 구분자: whitespace 전체 |
| readLine vs nextLine | 결과는 같음, 가져오는 방식이 다름 |
| 지역성 | 연속 메모리 순차 접근 → 캐시 히트율 향상 |
| 출력 | System.out과 BufferedWriter는 각각 별도 버퍼 |
