# Claude Code — 워크플로와 컴포넌트 쇼케이스

## 이 문서의 위치

[[development/vibe-coding/02.tools|02. 도구]]에서 Claude Code를 **도구**로 소개한다. 이 문서는 **실행 순서·프롬프트 예시·쇼케이스 작업**에 집중한다. 사전 기획·PRD·`CLAUDE.md` 작성 습관은 [[development/vibe-coding/06.idea|06. 아이디어와 사전 설계]]를 따르고, 시큐어 코딩·개인정보 등 **보안 원칙**은 [[development/vibe-coding/03.security|03. 보안]]을 따른다.

## 공식 문서

Claude Code의 설치·명령·업데이트는 **공식 문서**가 기준이다. 제품이 바뀔 수 있으므로 세부 CLI·슬래시 커맨드는 문서에서 확인한다.

- [Claude Code 문서(한국어 개요)](https://code.claude.com/docs/ko/overview)

## 프로젝트 컨텍스트(CLAUDE.md)

저장소 루트의 **`CLAUDE.md`**(또는 프로젝트가 정한 컨텍스트 파일)는 Claude가 **프로젝트를 이해하는 허브**다. 예시로 넣을 수 있는 항목:

- 프로젝트 구조·아키텍처 요약, 사용 **기술 스택**
- 코딩 컨벤션·금지 사항
- 보안 관련 주의(상세는 [[development/vibe-coding/03.security|03]])
- 자주 쓰는 작업에 대한 짧은 지시
- **응답 언어:** 한국어로 답하도록 요청할 경우 명시
- **GitHub** 계정·원격 저장소 URL, 기본 **작업 디렉터리**

`init` 등으로 초기 컨텍스트를 채우는 절차는 공식 문서를 따른다. PRD·Roadmap·중간 요약 md 전략은 [[development/vibe-coding/06.idea|06]]와 맞춘다.

## 권장 워크플로

| 단계 | 하는 일 | 예시 프롬프트 |
|------|---------|----------------|
| 탐색 | 프로젝트 구조·목적 파악 | 이 프로젝트가 무엇을 하는지 요약해 줘. |
| 계획 | 구현 방향·트레이드오프 논의 | 로그인 기능을 JWT로 할지 세션으로 할지 검토해 줘. |
| 구현 | 코드·설정 생성 | JWT 기반 로그인 API를 프로젝트 스타일에 맞게 구현해 줘. |
| 검증 | 테스트·동작 확인 | 방금 만든 API용 단위/통합 테스트를 추가해 줘. |
| 커밋 | 버전 관리 | 변경 사항을 커밋 메시지 규칙에 맞게 커밋해 줘. |

## Plan 모드와 UI 작업

길게 설계하거나 여러 파일에 걸친 UI 작업을 할 때는 **Plan 모드**(제품 UI에서 제공되는 경우)로 방향을 잡은 뒤, 구현 지시를 이어 가면 맥락이 덜 끊긴다. Plan·Desktop·Code의 **역할 감각**은 [[development/vibe-coding/06.idea|06]]의 「설계에 쓰는 도구」를 참고한다.

## 컴포넌트 쇼케이스(테스트 페이지)

원하는 UI를 **한곳에서** 검증하려면 앱 안에 **컴포넌트 쇼케이스**(디자인 시스템 프리뷰·스타일 가이드 페이지)를 두는 방식이 흔하다. 프레임워크는 프로젝트에 맞게 선택한다([[development/vibe-coding/01.prologue|01]]의 Next 등).

**권장 순서(요약)**

1. 쇼케이스에 올릴 **컴포넌트 목록**과 **테마·팔레트**를 프롬프트에 담는다. 스타일을 나중에 **일괄 변경**할 수 있게 토큰·테마 구조를 요청한다.
2. 필요하면 **Plan 모드**로 페이지 구조·파일 배치를 먼저 잡는다.
3. 생성된 결과를 보며 일부를 직접 수정해 감을 맞춘다.
4. 확정한 토큰·규칙은 **문서나 설정에 기록**해 두어 [[development/vibe-coding/06.idea|06]]의 디자인 기준과 맞춘다.

**쇼케이스에서 정할 것**

- **프리뷰만** 보여 줄지, **사용 예시 코드 스니펫**(복사용 JSX 등)까지 노출할지.
- **프리셋 테마:** UI 라이브러리가 제공하는 테마를 고르는 경우와, 팀이 토큰 묶음을 직접 정의하는 경우가 모두 있다. 둘 다 “미리 짜인 조합을 선택·확장”한다는 점에서 프리셋에 가깝다.
- **대안:** Storybook 등 전용 도구로 둘지, 앱 라우트(예: `/ui`)로 둘지 팀 규모에 맞게 선택한다.

**컴포넌트 예시(카테고리)**

전부 필수는 아니다. 프로젝트에 맞게 줄인다.

- **피드백:** Spinner, Skeleton, Toast, Modal, Badge  
- **폼·입력:** Button, Input, Dropdown  
- **레이아웃:** Container, Stack, Header/Footer  
- **데이터 표시:** Card, Table, Pagination, Avatar  

일관된 UI를 위해 **스타일을 먼저 정의**하고, 쇼케이스에서 수정한 내용을 **최종적으로 문서·토큰에 반영**한다.

## 화면 검증

쇼케이스·실서비스 화면의 DOM 구조·클래스는 **Chrome 개발자 도구**로 확인한다. [[development/vibe-coding/06.idea|06]]의 벤치마킹은 **남의 서비스**를 볼 때, 이 절은 **내가 만든 UI**를 검증할 때에 가깝다. `id`·`class`로 요소를 식별할 수 있다.
