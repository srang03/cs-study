# 관리자 시스템 — 역할과 기능 범위

## 이 문서의 위치

[[development/vibe-coding/04.service-operation|04. 서비스 운영]]에서 호스팅·비용·운영 모델을 정한 뒤, 실제 서비스를 돌리면서 **관리자 화면으로 무엇을 다룰지**를 이 문서에서 요구사항 수준으로 정리한다. 네트워크·WAF·시크릿 등 **인프라 보안**은 [[development/vibe-coding/03.security|03. 보안]]에 맡긴다.

## 관리자 대시보드의 역할

운영·고객 대응·콘텐츠·권한 통제를 **한 허브**에서 처리하기 위해 관리자 전용 UI(대시보드)가 필요하다. 기능 범위는 서비스 성격에 따라 줄이거나 늘리면 된다.

## 계정 유형과 접근 통제(RBAC)

기능 요구사항 중 **계정 유형과 권한 모델**이 가장 먼저 고정되는 편이 안전하다.

- **역할 예시:** 일반 사용자, 중간 관리자, 최고 관리자 등. 이후 **역할이 늘어날 여지**를 설계에 남긴다.
- **접근 통제:** 역할(Role)과 권한(Permission)을 분리해 두면, 같은 “관리자”라도 메뉴별로 다르게 열 수 있다. **역할 중복·겹침**(한 사람에게 여러 역할)도 정책으로 정한다.
- **라우트:** 일반 사용자는 **관리자 URL·API**에 접근하지 못하게 막고, 관리자 전용 페이지는 **인증된 관리자만** 통과시킨다. 구현은 서버 측 검증을 전제로 한다.
- **감사(audit):** 환불·계정 차단·중요 설정 변경 등은 **누가·언제** 수행했는지 남기면 운영·분쟁 대응에 유리하다.
- **역할×기능 표:** 세부 매트릭스는 문서가 길어지므로 **구현 단계에서 표로 정리**해도 된다.
- **관리자 계정 보호:** 가능하면 **2단계 인증(2FA)** 등을 관리자 계정에만 우선 적용하는 것을 권장한다(일반 로그인 UX와는 별도로 검토).

## 기능 영역(예시)

아래는 **예시 목록**이다. 실제 메뉴는 서비스에 맞게 합치거나 나눈다.

- **도메인 관리:** DNS·인증서·도메인 연결 상태 등 운영에 필요한 정보(세부는 호스팅 환경에 따름).
- **회원:** 전체 회원 목록, 역할이 다른 계정(예: 강사·관리자) 구분·검색.
- **결제·주문:** 개별 구매 내역(주문 번호, 구매자, 강의/상품, 결제 수단·금액), **결제 실패** 처리, **환불** 흐름. 결제 관련 상세는 이 블록에만 둔다.
- **쿠폰·할인:** 강의 구매 할인, 적용 범위·기간, 쿠폰 코드, 정책·마케팅 할인.
- **리뷰:** 사용자 리뷰 목록, **악성·분쟁 리뷰** 대응, 필요 시 사용자 측과의 소통 이력.
- **이벤트·공지·배너:** 공지사항, 이벤트 페이지, 메인 배너 클릭 시 안내 페이지 연결.
- **통계·분석:** 운영 지표(매출, 가입, 이탈 등)는 정의에 따라 범위를 좁히거나 외부 BI로 넘길 수 있다.
- **시스템·관리자 설정:** 전역 플래그, 관리자 계정·역할 할당 등 **메타 설정**. 콘텐츠·노출에 가까운 항목은 다음 절.
- **알림:** 채널·인프라 선택은 아래 **「알림(별도 구성 권장)」** 절을 본다.
- **세션·로그인:** 운영 정책은 아래 **「세션·로그인 운영」** 절을 본다.

## 세션·로그인 운영

- **세션 타임아웃:** 관리자·일반 사용자 각각 정책을 둘 수 있다.
- **로그인 기록:** 접속 IP, User-Agent, 기기/브라우저 식별(가능한 범위) 등. 이 항목은 **개인정보·로그 보관**과 맞닿으므로 수집 범위·보관 기간은 [[development/vibe-coding/03.security|03. 보안]]의 개인정보 최소화 원칙과 맞춘다.
- **활성 세션:** 동시 로그인(여러 기기) 허용 여부, 원격 로그아웃.
- **차단 계정:** 제재·탈퇴·정지 계정의 재로그인 방지 및 해제 절차.

## 콘텐츠·마케팅 설정

- **강의 카테고리** 구조.
- **인기 검색어:** 검색어 자체와 **노출 순서**.
- **외관(마케팅):** 시스템 로고, 메인 배너 등 랜딩과 직결되는 요소.

## 알림(별도 구성 권장)

알림은 기능이 커지기 쉬워 **도메인을 나누는 것**을 권장한다.

- **전달 방식:** 주기적 폴링, **웹소켓** 등 실시간 채널 선택.
- **인앱 알림:** 할인 쿠폰 도착, 수강 안내 등 **주기적·이벤트성 안내**.
- **이메일:** 발송용 서비스(SendGrid, AWS SES 등) **별도 가입·설정**이 필요하다. AWS SES 등은 **도메인·발신 검증** 절차가 있을 수 있다.
- **운영 통계·알림 설정:** 알림 자체의 통계·재시도·실패 큐를 둘지, **별도 마이크로서비스/큐**로 뺄지는 트래픽과 팀 규모에 따라 **신중히** 결정한다.
