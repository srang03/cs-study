# Todo 예제로 이어가는 바이브 코딩 절차

## 이 문서의 위치

이 문서는 **Todo(+로그인·JWT 등 인증)** 를 예로 들어, AI와 설계·프롬프트를 정리한 뒤 구현까지 이어 갈 때의 **실행 순서·체크리스트**에 집중한다.

- **다루지 않는 것:** 사전 설계 원칙, PRD·산출물 목록, SQL·DB 원칙의 상세는 [[development/vibe-coding/06.pre-implementation-design|06. 아이디어와 사전 설계]]를 따른다.
- **공통 워크플로·컴포넌트 쇼케이스:** [[development/vibe-coding/07.workflow-and-component-showcase|07. 워크플로와 컴포넌트 쇼케이스]]와 같은 권장 순서를 따르며, 여기서는 Todo 케이스에 맞게 단계만 압축해 정리한다.
- **보안·시큐어 코딩:** [[development/vibe-coding/03.vibe-coding-security|03. 보안]]을 따른다.
- **트랙 범위·스택 감각:** [[development/vibe-coding/01.prologue|01. 프롤로그]], 도구·환경은 [[development/vibe-coding/02.dev-tools|02. 도구]]를 참고한다.

**이 문서의 가정:** Todo CRUD에 더해 **인증(예: JWT)** 까지 포함한 **풀스택 학습·프롬프트 실습**을 목표로 한다. 인증 없이 Todo만 다룰 경우 아래 단계에서 해당 요구만 생략하면 된다.

**기술 스택(예시):** 이 트랙에서는 Next.js, **PostgreSQL**, 정적·프런트 배포로 **S3** 등, API·인프라로 **AWS** 를 염두에 둔 사례가 많다. UI는 **shadcn/ui** 같은 컴포넌트 세트를 쓰는 경우를 예로 들 수 있으나, 동일한 절차를 다른 스택에도 적용할 수 있다.

## Todo를 예제로 쓰는 이유

- **CRUD**가 한 번에 들어 있어 게시판 등 다른 도메인으로 확장하기 좋은 **축소 예제**에 가깝다(엄밀히 “게시판”과 동일하진 않다).
- **사용자 인증**을 붙이기 쉬워 로그인·토큰·권한을 연습하기 좋다.
- 범위를 통제하면 **간단한 웹 서비스**를 끝까지 경험하기에 적당하다.

## 전제: 환경과 구조

- **개발과 운영을 구분**한다. 로컬에서 개발하고, 배포·시크릿·DB는 운영 기준을 미리 염두에 둔다(호스팅·운영 모델은 [[development/vibe-coding/04.service-hosting-and-ops-model|04. 서비스 운영]] 참고).
- **프런트엔드와 백엔드(또는 API)를 분리**하는 편이 유지보수에 유리하다. UI·표현과 데이터·비즈니스 로직을 나누면 화면 변경이 API·도메인에 덜 미치고, [[development/vibe-coding/01.prologue|01]]에서 말한 **AWS 배포** 전환도 단계적으로 하기 쉽다.
- **HTTP API**는 통신마다 이전 상태를 서버가 기본적으로 들고 있지 않는 **stateless** 한 성질을 전제로 설계하는 경우가 많고, 인증은 **세션** 방식과 **JWT 등 토큰** 방식 등 선택지가 있다. 세부 원칙·주의는 [[development/vibe-coding/03.vibe-coding-security|03]]에서 정리한다.

## AI에게 구현을 맡기기 전 준비

**단순한 한 줄 지시**만으로는 결과가 흔들리기 쉽다. [[development/vibe-coding/06.pre-implementation-design|06]]에서 말한 것처럼 **문서·맥락을 파일로 쌓아 두고** 같은 기준으로 프롬프트를 만든다.

- 기획·목표 사용자·주요 기능·기술 스택, 페이지 구조·개발 절차, (가능하면) **레퍼런스 URL** 분석 요청 등.
- 설계 대화·회의 내용은 **세션·요약 md**로 남겨 나중에 이어 쓴다.

**케이스별로 채울 문서 세트 예시(한 블록으로 정리):**

| 구분 | 담을 내용(예) |
|------|----------------|
| 공통 | 프로젝트 개요·주요 기능, 기술 스택, 디렉터리·프로젝트 구조, 구현 우선순위 |
| 백엔드/API | 배포·인프라 개요, 의존성·런타임, API 경계 |
| 프런트엔드 | 정적 호스팅·**S3 버킷** 규칙(해당 시), 스타일(**Tailwind CSS** 등), 주의사항·완료 체크리스트 |

위 순서는 **기획 흐름**으로도 쓸 수 있다: 서비스 목적·범위 → 요구사항·기능 정의 → 기술 검토 → 스택·구성 확정 → 레퍼런스·UI 기획 → 디자인 시스템·컴포넌트 → 클라우드·배포 검토.

## 설계 산출 순서와 구현 순서

- **문서·프롬프트를 정리하는 순서**로는 **UI(프런트) 방향, DB 스키마, API·백엔드** 순이 이해하기 쉬운 경우가 많다. 화면과 데이터 모델이 잡히면 엔드포인트가 따라가기 쉽다.
- **실제 코딩 순서**는 팀·스택에 따라 달라도 되며, 중요한 것은 **같은 합의가 PRD·프롬프트·`CLAUDE.md`에 남는 것**이다.

## AI 설계 단계: 순서와 체크리스트

상세한 프롬프트 작성법·Plan 모드·쇼케이스 구성은 [[development/vibe-coding/06.pre-implementation-design|06]], [[development/vibe-coding/07.workflow-and-component-showcase|07]]를 병행한다.

| 단계 | 목적 | 하는 일 | 산출·메모 |
|------|------|---------|-----------|
| 1 | 맥락 고정·누락 보완 | 기술 스택과 **3계층(클라이언트·API·DB)** 이 드러나게 최초 프롬프트를 쓴다. AI 답변으로 빠진 제약을 채운다. | 기능은 대화로 구체화해도 된다. **아직 코드가 필요 없고 설계에 집중**하라고 명시한다(에이전트가 곧바로 코드를 쓰는 경향에 대비). |
| 2 | UI·컴포넌트 확정 | 스타일·테마 후보를 AI에게 제안받고 **하나로 결정**한다. 제안된 스택을 채택한다고 합의해 고정한다. | UI 컴포넌트 체계가 빠져 있으면 **반드시 요청**한다. 예: React 생태계에서는 **shadcn/ui** 등 컴포넌트 템플릿 모음을 포함하도록 지시. |
| 3 | 쇼케이스로 검증 | [[development/vibe-coding/07.workflow-and-component-showcase|07]]의 **컴포넌트 쇼케이스**를 만든 뒤 시각적으로 확인한다. | 빠진 컴포넌트·수정은 프롬프트로 보완. |
| 4 | 기능 명세 고정 | 예: 「Todo 목록 기능을 요구사항 수준으로 다시 정리해 줘」처럼 **도메인별로 재정리**한다. | API·화면·권한이 한글 설명으로 맞는지 확인. |
| 5 | DB 스키마 이름·스크립트 | **스키마 이름은 소문자** 등 팀 규칙을 AI에게 명시하고, **DBML** 또는 SQL 초안을 다시 쓰게 한다. | **스키마 생성·적용은 사람이 주도**한다. Claude Code 등에 **프로덕션 DB에 직접 접속**하게 맡기지 않는다(오류·보안 위험). 상세 원칙은 [[development/vibe-coding/06.pre-implementation-design|06]]의 SQL·DB 절. |
| 6 | 최종 합의·산출물 | 보완 사항·트레이드오프를 짚고 **사람이 결정**한다. | 아래 「최종 산출물」 체크리스트를 채운다. |

### 품질·보안·성능(단계 4~6 사이에 반영)

- **코딩 컨벤션·시큐어 코딩**을 프롬프트나 `CLAUDE.md`에 넣는다. 상세는 [[development/vibe-coding/03.vibe-coding-security|03]].
- **성능·비기능**(응답 시간, 페이지 크기 등)을 한 줄이라도 요구사항에 적어 둔다.

## 최종 산출물

구현 단계([[development/vibe-coding/07.workflow-and-component-showcase|07]]의 구현·검증 흐름)로 넘기기 직전에 다음을 갖춘다.

- [ ] **Claude Code(등)용 프롬프트**를 하나 이상의 `.md` 파일로 저장했다(세션 한도·재시작 대비).
- [ ] **DB 스키마 정의**(DBML·SQL)가 **스키마명·테이블 순서**와 맞고, 실행은 **로컬/스테이징에서 사람이 검토**한다.
- [ ] **`CLAUDE.md`**(또는 프로젝트 컨텍스트)에 스택, 디렉터리 규칙, 금지 사항, API·인증 방침 요약, 응답 언어 등 [[development/vibe-coding/07.workflow-and-component-showcase|07]]에서 예시로 든 항목을 반영했다.
